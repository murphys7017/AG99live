# 前后端严格审阅报告（AG99live）

快照日期：2026-04-22
审阅范围：`adapter/`（AstrBot 插件后端）+ `frontend/`（Electron/Vue）+ 现有项目文档 + AstrBot 文档
审阅方式：代码静态审阅 + 运行验证（`pytest`、`typecheck`）

## 结论摘要

当前链路可运行，但在“动作计划实时生成”进入主链路后，存在 1 个高优先级时序风险和 2 个中高优先级结构问题。它们不会让系统立刻崩溃，但会在并发对话或对外暴露调试端口时带来错误动作投递与安全隐患。

## 运行验证

- 后端：`python -m pytest adapter/tests -q` -> `16 passed`
- 前端：`npm run typecheck` -> 通过

## 发现（按严重级别）

### P0 - 实时动作计划可能串到错误轮次（stale turn）

证据：
- `adapter/adapter/turn_coordinator.py:466-471` 在异步任务启动时只捕获 `user_text/assistant_text`，未绑定原始 `turn_id`。
- `adapter/adapter/turn_coordinator.py:444` 广播时使用的是 `self.session_state.current_turn_id`（当前轮），不是生成该计划时的轮次。
- `frontend/src/composables/useAdapterConnection.ts:854-877` 前端收到 `engine.motion_plan` 后立即生效，未按 `turn_id` 做轮次隔离。

影响：
- A 轮文本生成慢、B 轮先开始时，A 轮动作可能被错误标记成 B 轮并立即执行。
- 用户感知为“动作和回复语义对不上”。

建议：
- 异步任务创建时显式捕获 `origin_turn_id`，广播时强制使用该值。
- 前端消费 `engine.motion_plan` 时增加 `turn_id` 守卫（只接受 `turn_id == currentTurnId` 或“无 active turn 时”策略）。

### P1 - 单次请求并链路的关键字段已被丢弃

证据：
- `adapter/platform_adapter.py:290-302` 仍然向下传了 `inline_base_expression/inline_motion_id`。
- `adapter/adapter/turn_coordinator.py:161-166` 进入协调器后直接 `del inline_base_expression` 和 `del inline_motion_id`。
- `adapter/platform_event.py:15-20` 事件层 `send()` 只传 `message_chain` 和 `unified_msg_origin`，未注入 inline 字段。

影响：
- 如果你要把“文本+动作元信息”合并到一次主请求，目前没有可用的承载通路。

建议：
- 二选一：
1. 恢复 inline 字段通道（事件层 -> 协调器 -> 协议下发）。
2. 在 `reply_text` 内使用结构化标签（如 `<@anim {...}>`）并在协调器中解析（更贴近现有实现）。

### P1 - Debug API 无鉴权 + 宽 CORS，存在误触发风险

证据：
- `adapter/platform_adapter.py:55` 默认开放 `debug_port=12398`。
- `adapter/platform_adapter.py:380-430` `/api/engine/motion_plan_preview` 接口仅校验 `plan` 为对象，无鉴权。
- `adapter/static_resources.py:44-47` 返回 `Access-Control-Allow-Origin: *`、允许 `POST`。

影响：
- 在 `host` 配置为非本地回环时，外部或同机网页脚本可直接触发动作计划注入。

建议：
- 默认仅监听 `127.0.0.1`（已是默认值），并新增可选 `debug_token` 或 header 校验。
- 至少为 debug API 增加“开关默认关闭 + 明确告警日志”。

### P2 - `engine.motion_plan` 入站缺少协议级 schema 校验

证据：
- `adapter/adapter/protocol.py:259-296` 的 `_validate_payload()` 未覆盖 `TYPE_ENGINE_MOTION_PLAN`。
- `adapter/adapter/turn_coordinator.py:405-423` 当前仅记录日志并返回（占位），未做更细粒度结构校验。

影响：
- 入站计划结构异常时，错误在前后端间延迟暴露，排障成本高。

建议：
- 在协议层补充 `engine.motion_plan` payload 最小 schema（`mode`, `plan.steps[]` 基本类型）。
- 对校验失败统一返回 `control.error`，避免静默吞掉问题。

### P2 - 文档状态与当前代码存在漂移

证据：
- `README.md:21-22`、`docs/V2当前实现状态与下一步.md:26,39,63` 仍描述 `engine.motion_plan` 仅测试占位。
- 但代码中已存在自动实时生成链路：`adapter/adapter/turn_coordinator.py:186,458-509`。

影响：
- 团队成员会误判线上行为（以为不会自动下发动作）。

建议：
- 更新文档为“占位执行器未完成，但已存在实时计划广播实验路径”。

### P3 - 测试覆盖不平衡，关键路径缺回归

证据：
- 当前单测集中在 `adapter/tests/unit/live2d/*` 与 `adapter/tests/unit/runtime/*`。
- 缺少 `turn_coordinator`、`protocol(engine.motion_plan)`、`debug API`、前端 `engine.motion_plan` 轮次守卫相关测试。

影响：
- 时序和协议回归更容易在联调/线上暴露。

建议：
- 增加 4 组回归：
1. stale-turn 防串轮测试。
2. `engine.motion_plan` schema 校验测试。
3. debug API 鉴权/开关测试。
4. 前端 `turn_id` gating 测试。
