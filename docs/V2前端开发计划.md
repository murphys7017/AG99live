# V2 前端开发计划

本文用于记录 AG99live V2 阶段中 `frontend` 侧的开发设想与初期计划。

当前阶段的目标不是直接展开页面开发，而是先明确前端在 V2 中真正承担的核心角色，并把研发重点从“普通前端页面”转向“本地 Live2D runtime + 动作引擎”。

## 1. 前端在 V2 中的定位

V2 前端不是单纯的页面层，而是：

- Live2D 渲染承载层
- 实时动作引擎执行层
- 本地用户交互层

如果只用一句话定义：

**V2 前端的本体不是 UI，而是本地 Live2D runtime。**

## 2. 当前已经明确的前端判断

### 2.1 Live2D 渲染很重要，但不是最难点

前端当然必须完成 Live2D 的渲染、加载、切换和基本播放能力。

但当前判断是：

- 这部分虽然重要
- 也需要工程实现
- 但相对来说并不是 V2 最难的部分

原因是：

- 相关资料相对充分
- 现有项目和开源社区中已有不少参考
- 当前 V1 中也已经有可迁移经验

因此，V2 前端不应把主要研发精力消耗在“单纯把模型显示出来”这件事上。

### 2.2 真正最难的是动作引擎

当前已经明确，V2 前端最难、也最关键的部分是 `engine`，也就是动作引擎。

动作引擎需要处理的不是单纯播放，而是：

- 接收后端给出的基础信息和语义信息
- 决定当前回复应该如何表现
- 组合参数化动作
- 与音频时长、节奏、强调点联动
- 在参数驱动、expression、motion fallback 之间做运行时调度

这意味着，前端真正的核心研发对象是：

**如何构建一套可实时执行的 Live2D 动作系统。**

## 3. 前端建议拆成的三部分

当前阶段，前端建议明确拆成三块：

1. 渲染层
2. 动作引擎层
3. UI 交互层

### 3.1 渲染层

职责包括：

- Live2D 模型加载
- 资源切换
- 画面渲染
- expression / motion 的基础播放接口
- 参数写入通道
- 音频与口型同步所需的渲染配合

这一层的目标是“提供可控模型”，而不是直接决定模型怎么动。

### 3.2 动作引擎层

这是 V2 前端的真正核心。

职责包括：

- 接收后端提供的动作相关输入
- 生成运行时动作计划
- 组合参数通道
- 控制强度、时长、节奏
- 在参数驱动和 fallback 动作之间切换
- 输出给渲染层执行

这一层后续大概率会成为 AG99live 最核心的技术资产。

### 3.3 UI 交互层

当前阶段对 UI 的判断是：

- 它重要
- 但当前不是最优先突破口
- 具体视觉设计可以后置

现阶段只需要保留最核心的桌宠交互能力即可。

## 4. 当前阶段对 UI 的设想

当前已经明确：

- 先保留桌宠模式
- 其他复杂 UI 暂时不作为最优先事项
- 对话历史、完整操作面板等可以保留后续讨论空间
- 具体 UI 视觉设计和页面细节后面再单独讨论

也就是说，V2 启动阶段不以“先把完整客户端界面做出来”为目标，而是以“先把桌宠运行时和动作引擎做出来”为目标。

## 5. 前端与后端的数据协作方式

当前阶段已经讨论到一个关键问题：

- 动作相关信息究竟是前端再次请求，还是由后端在回复时一并给出

当前倾向是：

**优先采用“后端在同一次回复中一并给出基础动作相关信息”的方式，而不是让前端为同一句回复再额外发起一次请求。**

这样做的好处是：

- 减少一次额外往返
- 降低时序复杂度
- 更有利于音频、文本、语义信息同步进入前端引擎
- 更适合桌宠场景的实时体验

但这里需要特别强调：

后端即使一并给出动作相关信息，也不意味着后端直接生成最终动作细节。

更合理的分工是：

- 后端提供低频输入
  - 例如语义倾向、情绪倾向、基础标签、模板线索
- 前端动作引擎负责生成最终实时表现

因此，这里的“后端一并给出”更接近：

**后端给前端动作引擎提供同包输入，而不是替代前端动作引擎本身。**

## 6. 当前阶段不着急解决的问题

以下问题当前可以暂时不在第一阶段细化：

- 完整 UI 视觉设计
- 完整对话历史面板设计
- 各种模式的最终交互布局
- 很细的协议字段结构
- 多模型完整兼容策略的最终版本

这些都很重要，但当前不是前端启动阶段的主要阻塞点。

## 7. 前端当前阶段的主要研发目标

V2 前端第一阶段更适合聚焦以下目标：

- 跑通一个可控的 Live2D 渲染基础
- 确认参数驱动通路可行
- 跑通一个最小动作引擎原型
- 验证“后端给语义输入，前端生成实时动作”的主链路
- 先服务桌宠模式，不追求完整 UI

## 8. 当前阶段的结论

当前已经确定的前端方向如下：

- 前端中最重要但不是最难的是 Live2D 渲染
- V2 前端真正最难、也最关键的是动作引擎
- 前端应拆成渲染层、动作引擎层、UI 层
- 当前阶段 UI 只保留桌宠模式即可
- 具体 UI 设计暂不作为当前重点
- 动作相关输入更适合由后端在同一次回复中一并提供
- 但最终实时动作生成仍应由前端动作引擎完成

一句话总结：

**V2 前端的首要任务不是做界面，而是把 Live2D 动作引擎跑起来。**
