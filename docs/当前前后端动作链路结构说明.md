# 当前前后端动作链路结构说明

本文描述 AG99live 当前与“文本 / 音频 / 动作”播放链路相关的前后端结构。目标不是讲历史方案，而是说明现在代码里每个核心模块、核心函数分别负责什么。

本文基于当前代码状态编写，适合作为后续继续开发、补测试、排查问题时的结构手册。

## 1. 当前结构评估结论

结论：**当前结构不需要继续做大规模拆分或重构。**

原因：

1. 主链路已经形成清晰分层：
   - 后端 `TurnCoordinator` 负责轮次编排、协议出入站、动作生成策略切换
   - 前端 `useAdapterConnection` 负责协议连接与状态汇聚
   - 前端 `useModelEngine` 负责动作载荷排队、等待音频、编译与起播
   - 前端 `usePlaybackCompletionCoordinator` 负责“文本 / 音频 / 动作”完成判定
   - 前端 `usePetDesktopController` 负责把各模块拼装成桌宠运行时
2. 最近几次重构已经把大量视图逻辑、连接逻辑、动作逻辑拆出，职责边界比之前稳定很多。
3. 现阶段剩下的主要风险不在“模块还不够拆”，而在：
   - 时序回归
   - 状态机边界
   - 缺少针对播放完成编排的前端自动化测试

因此，**下一阶段更值得投入的是补测试、补协议文档、补时序用例，而不是继续拆结构。**

只有两个可以保留为“以后再优化”的候选点，但都不属于当前必须做：

- `frontend/src/composables/useAdapterConnection.ts`
  这个文件仍然偏大，但它现在已经是“连接层总装配器”，不是杂糅式巨石；继续拆收益有限。
- `frontend/src/composables/usePetRuntimeSnapshotPublisher.ts`
  现在主要是一个大 watch 汇总器，可读性一般，但职责单一，没有明显架构风险。

## 2. 当前主链路总览

### 2.1 文本输入到播放完成的链路

1. 前端通过 `useAdapterConnection.sendText()` 发送 `input.text`
2. 后端 `TurnCoordinator.handle_msg()` 接收入站消息
3. 后端 `_commit_inbound_message()` 建立新 turn，并发送 `control.turn_started`
4. 后端大模型回复后，`emit_message_chain()` 输出：
   - `output.text`
   - 可选 `engine.motion_intent` / `engine.motion_plan`
   - 可选 `output.audio`
   - `control.synth_finished`
5. 前端 `useAdapterConnection.handleSocketMessage()` 分发这些消息并写入状态
6. 前端 `useModelEngine.ingestInboundPayload()` 接收入站动作载荷，必要时等待音频起播后再开始动作
7. 前端 `usePreviewMotionPlayer.playPlan()` 真正驱动 Live2D 动作
8. 前端 `usePlaybackCompletionCoordinator` 观察文本、音频、动作三类状态
9. 当前端确认三者都完成后，如果需要，就回发 `control.playback_finished`
10. 后端 `TurnCoordinator.finalize_turn()` 真正结束该轮，并发送 `control.turn_finished`
11. 前端收到 `control.turn_finished` 后做最终清理

### 2.2 当前一轮结束的判定

当前实现下，一轮的“播放完成”不是单看音频结束，而是按下面逻辑处理：

1. 文本已送达
2. 音频状态已明确
   - `completed`
   - `failed`
   - 或无音频时走 `not_requested`
3. 动作若预期会来，则必须等动作完成
4. 对“音频先结束，动作可能晚到”的情况，会有一个短暂 settlement window
5. 满足条件后，前端决定是否发送 `control.playback_finished`
6. 后端收到该信号后发送 `control.turn_finished`
7. 前端在收到 `control.turn_finished` 后，才做该组 turn/orchestration 的最终清理

## 3. 前端模块与函数职责

## 3.1 `frontend/src/composables/usePetDesktopController.ts`

作用：桌宠运行时的总装配器。它不直接实现协议、动作编译或播放，而是把这些能力组合起来。

### `usePetDesktopController()`
- 创建并连接：
  - `useAdapterConnection`
  - `useDesktopBridge`
  - `useModelSync`
  - `usePreviewMotionPlayer`
  - `useModelEngine`
  - `usePlaybackCompletionCoordinator`
- 负责桌面命令分发
- 负责把 adapter 状态送进 model engine
- 负责把 runtime snapshot 发布到 Electron bridge

### `applyMotionEngineSettingsSnapshot(nextValue)`
- 将 bridge snapshot 中的动作引擎设置同步到本地响应式对象
- 做归一化和相等判断，避免无意义覆盖

### `applyAmbientMotionPreference(attemptsRemaining = 12)`
- 尝试把“环境动作开关”应用到 Live2D 运行时
- 如果运行时还没挂好，则重试若干次

### `handlePreviewMotionPlan(plan)`
- 本地直接预览动作载荷
- 同时把预览载荷通过 adapter 发回后端，用于动作实验室或调试链路记录

### `saveMotionTuningSample(sample)`
- 保存动作调优样本到后端

### `deleteMotionTuningSample(sampleId)`
- 删除动作调优样本

### `handleDesktopCommand(command)`
- 桌面主窗口收到 bridge 命令后的统一入口
- 负责连接、断开、发文本、打断、预览动作、开关麦克风、历史操作、调优样本操作等

### `handleProfileAuthoringCommand(command)`
- 处理 Profile Editor 发来的主轴配置保存请求

### `showContextMenu(event)`
- 请求 Electron 宿主弹出右键菜单

## 3.2 `frontend/src/composables/useAdapterConnection.ts`

作用：前端协议连接层总入口。它负责 WebSocket、协议消息分发、前端状态聚合，以及把不同方向的能力委托给更小的模块。

### `resetAudioPlaybackTerminal()`
- 重置音频终态状态组

### `sendMicrophoneAudioChunk(chunk)`
- 将麦克风采集到的音频片段通过 socket 发给后端
- 在连接未建立或缓冲过高时丢弃

### `persistAddress(nextAddress)`
- 归一化后保存 adapter 地址

### `setDesktopScreenshotOnSendEnabled(enabled)`
- 更新“发送文本时附带桌面截图”开关并持久化

### `setAddress(nextAddress)`
- 修改 adapter 地址

### `initialize()`
- 初始化连接层的持久化配置

### `connect()`
- 进入连接流程
- 生成候选地址并调用 `openConnectionCandidate`

### `disconnect()`
- 主动断开连接

### `disconnectInternal(markManualClose)`
- 内部断开实现
- 负责停麦克风、停音频、关闭 socket

### `openConnectionCandidate(candidates, index, attemptSerial)`

### `resetConnectionRuntimeState()`

### `toggleMicrophoneCapture()`
- 在开启 / 关闭麦克风之间切换

### `startMicrophoneCapture()`
- 请求浏览器麦克风权限并启动采集

### `stopMicrophoneCapture(reason = "manual_stop")`
- 停止麦克风采集
- 必要时向后端发送 `input.mic_audio_end`

### `handleSocketMessage(rawData)`
- WebSocket 入站消息统一分发器
- 校验协议版本
- 按 `type` 路由到各个 `apply*` 函数

### `applyServerInfoMessage(envelope)`
- 处理 `system.server_info`
- 同步后端地址、自动麦克风等运行时信息

### `applyOutputText(envelope)`
- 处理 `output.text`
- 更新当前 turn/orchestration、文本状态、助手回复文本

### `applyOutputAudio(envelope)`
- 处理 `output.audio`
- 若带文字也会顺带更新文本状态
- 若有音频地址则触发音频播放

### `applyOutputImage(envelope)`
- 处理 `output.image`
- 记录本轮图片数量

### `applyOutputTranscription(envelope)`
- 处理 `output.transcription`
- 记录语音转写文本

### `applyTurnFinished(envelope)`
- 处理后端发来的 `control.turn_finished`
- 更新 turn finished 状态
- 在无音频场景下补齐 `not_requested`

### `applyControlError(envelope)`
- 处理 `control.error`
- 写入错误态与历史记录

### `applySemanticAxisProfileSaved(envelope)`
- 处理主轴配置保存成功消息

### `applySemanticAxisProfileSaveFailed(envelope)`
- 处理主轴配置保存失败消息

### `reportProtocolError(key, message, envelope?)`
- 报告协议错误
- 去重同类错误，避免历史刷屏

### `reportUnhandledInboundEnvelope(envelope)`
- 报告前端尚未接入的入站协议消息

### `updateAssistantText(text, turnId)`
- 更新最后一条助手文本
- 对同一 turn 的重复文本做去重，避免历史列表重复

### `playAudioAndAcknowledge(audioUrl, turnId)`
- 连接层对音频播放模块的包装调用

### `stopAudioPlayback()`
- 停止当前音频播放

### `sendText(text)`
- 发送文本消息到后端
- 这是对 `outboundActions.sendText` 的包装

### `interruptCurrentTurn()`
- 向后端发送中断请求

### `sendSemanticAxisProfileSave(payload)`
- 发送主轴配置保存请求

### `requestHistoryList(options)`
- 请求后端历史摘要列表

### `createHistory(options)`
- 请求后端新建历史

### `loadHistory(historyUid, options)`
- 请求后端加载某个历史

### `deleteHistory(historyUid, options)`
- 请求后端删除某个历史

### `saveMotionTuningSample(sample)`
- 请求保存动作调优样本

### `deleteMotionTuningSample(sampleId)`
- 请求删除动作调优样本

### `sendMotionPayloadPreview(payload)`
- 发送动作测试载荷预览

### `sendMotionPlanPreview(plan)`
- 与 `sendMotionPayloadPreview` 同义的兼容入口

### `sendPlaybackFinished(turnId, orchestrationId, success, reason?)`
- 回发 `control.playback_finished`
- 告诉后端当前这一组播放已经完成

### `clearPlaybackGroupContext(turnId, orchestrationId)`
- 清理某一组播放上下文
- 只清理匹配当前组或匹配对应音频终态 / 起播状态的内容
- 用来避免旧轮次误清新轮次

### `rewriteModelSyncEnvelope(envelope)`
- 用当前活跃 host 改写 model sync 消息中的地址

### `rewriteSocketUrl(rawUrl)`
- 改写 socket URL

### `rewriteHttpUrl(rawUrl)`
- 改写 HTTP URL

### `pushHistory(role, text)`
- 追加前端本地历史记录

### `useAdapterConnection()`
- 对外导出整个连接层能力
- 负责创建 history adapter 与 motion tuning adapter

## 3.4 `frontend/src/adapter-connection/audioPlaybackActions.ts`

作用：纯音频播放动作函数集合。

### `playAudioAndAcknowledge(ctx, audioUrl, turnId)`
- 启动音频播放
- 在不同回调中记录：
  - 音频起播
  - 时长更新

## 3.5 `frontend/src/composables/usePlaybackCompletionCoordinator.ts`

作用：前端播放完成编排层。它不负责播放本身，而是负责判断一轮什么时候真的可以算完成。

### `usePlaybackCompletionCoordinator(options)`
- 创建并维护一组播放协调状态
- 监听文本、音频、动作、turn finished 的变化

### `clearPlaybackSettlementTimer()`
- 清理“等待晚到动作”的 settlement timer

### `resetPlaybackCoordination()`
- 重置本地播放协调状态

### `ensurePlaybackCoordination(turnId, orchestrationId)`
- 切换到新的 turn/orchestration 组时重置旧协调状态

### `shouldSendPlaybackFinished()`
- 判断当前是否满足“可宣布播放完成”

### `isPlaybackAckRequired()`
- 判断当前是否需要真的向后端回发 `control.playback_finished`
- 只有音频为 `completed` 或 `failed` 时才需要

### `finalizePlaybackCoordination()`
- 在前端侧真正清理该播放组上下文

### `maybeFlushPlaybackCompletion(reason?)`
- 核心收敛函数
- 根据当前状态决定：
  - 是否发送 `playback_finished`
  - 是否等待后端 `turn_finished`
  - 是否立即做前端清理

### `schedulePlaybackSettlementWindow()`
- 在音频结束但动作可能晚到时，开启短暂等待窗口

### `recordMotionPlayback(event)`
- 记录动作起播历史
- 供动作实验室、历史视图、调试链路使用

## 3.6 `frontend/src/model-engine/useModelEngine.ts`

作用：前端动作引擎编排层。负责入站动作载荷的排队、等待音频、编译、启动播放。

### `syncPendingState()`

### `clearPendingPayload(entry)`

- 清理全部待处理动作

### `ingestInboundPayload(payload, context)`

### `notifyAudioPlaybackStarted(turnId)`

- 这通常会触发对应动作起播

### `notifyCurrentTurnChanged(turnId)`

### `playPreviewPayload(payload)`
- 本地立即预览动作载荷

### `stop(reason = "stopped")`
- 停止动作引擎
- 会清空 pending 队列并停止播放

作用：后端编排中枢。它负责协议入站、turn 生命周期、消息输出、动作策略切换、播放结束收口。

### `__init__(...)`
- 注入所有依赖
- 创建语音入站服务
- 初始化 turn lock、后台任务集合、时序统计容器

### `handle_msg(raw_message)`
- 后端协议入站总入口
- 负责分发：
  - `system.*`
  - `control.playback_finished`
  - `control.interrupt`
  - 各类音频输入
  - `input.text`
  - `engine.motion_*` 预览
  - 未接入类型的 `control.error`

### `emit_message_chain(message_chain, unified_msg_origin=None, inline_base_expression=None, inline_motion_id=None, raw_reply_text_override=None)`
- 将 AstrBot 产出的消息链转成前端协议输出
- 负责：
  - 提取文本、图片、音频
  - 处理 inline motion
  - 根据模式决定是否即时广播动作或回退到二次生成
  - 发送 `output.text / output.image / output.audio / control.synth_finished`
  - 无音频时直接结束 turn

### `finalize_turn(turn_id, success=True, reason=None)`
- 处理前端回发的 `control.playback_finished`
- 校验是否属于当前轮次
- 然后调用 `_finish_turn`

### `_commit_inbound_message(message_obj, turn_id=None)`
- 将用户入站文本真正提交为新 turn
- 负责：
  - 如上一轮还在等待播放完成，则先 finalize
  - 开始新 turn
  - 发送 `control.turn_started`
  - 构造并提交平台事件
  - 根据模式注入 inline motion contract 或禁用 streaming

### `_apply_inline_motion_contract_to_event(event, message_obj)`

### `_emit_image_input_diagnostics(message_obj)`

### `_describe_image_input_reason(reason)`

## 5. 当前结构下的开发建议

当前不建议继续大拆结构，建议按下面顺序推进：

1. **补前端时序测试**
   - `playback completion`
   - `late motion`
   - `interrupt during playback`
   - `turn switch while old settlement window still pending`
2. **补协议状态图**
   - 明确 `turn_started -> output.* -> synth_finished -> playback_finished -> turn_finished`
3. **补集成级链路验证**
   - 文本 only
   - 文本 + 音频
   - 文本 + 音频 + 动作
   - 动作晚到
   - 音频失败

如果后续需要继续优化，优先方向也不是再拆文件，而是：

- 给前端连接层补单测夹具
- 给播放完成编排层补状态机测试
- 给后端 `TurnCoordinator` 再补几条协议异常 / stale turn / interrupt 用例
