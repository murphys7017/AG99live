# V2 消息适配审阅与进度

快照日期：2026-04-21  
范围：`adapter` 消息适配层（不含真实动作引擎执行）

## 结论

- V2 协议主链路已落地，可用于前后端联调与日常对话链路。
- 传输层已稳定为 `WebSocket(实时)` + `HTTP(静态资源)`。
- 用户侧保持单地址输入，内部自动派生 WS/HTTP 地址。
- `engine.*` 当前只到协议层和测试入口，未作为生产动作接口。

## 协议总览

### 统一 envelope

所有消息统一字段：

- `type`
- `version`
- `message_id`
- `timestamp`
- `session_id`
- `turn_id`
- `source`
- `payload`

### 大类状态

- `input`：已接入
- `output`：已接入
- `control`：已接入
- `system`：已接入
- `engine`：骨架已接入（`motion_plan` 仅预览占位）

### `engine.*`（本轮状态）

- 已定义：`engine.motion_plan`、`engine.expression_plan`、`engine.idle_state`、`engine.look_target`
- 已接入：`engine.motion_plan` 入站允许 + turn coordinator 占位处理
- 未接入：真实动作规划执行

## 生命周期（当前实现）

1. 前端发送 `input.*`
2. adapter 回复 `control.turn_started`
3. adapter 发送若干 `output.*`
4. 有音频时发送 `control.synth_finished`
5. 前端播放完回传 `control.playback_finished`
6. adapter 回复 `control.turn_finished`

## 架构职责

1. `transport_ws`：连接、收发、envelope 级校验
2. `protocol`：消息类型定义、入站合法性校验、标准化
3. `turn_coordinator`：按类型分发主业务链路
4. 业务模块：语音入口、历史桥接、媒体缓存、系统同步

## 本轮新增进展

- 协议对 `engine.motion_plan` 入站放行，便于动作实验室联调。
- 文本/音频/图片回复已拆成独立 `output.*`，不再混合单消息承载。
- `control.error` 用于非法消息或链路错误反馈。

## 已验证

- 文本对话可闭环（`input.text` -> `output.text`）
- 音频链路可闭环（输入流、转写、播放完成回传）
- 历史、背景、心跳等 `system.*` 链路可用
- 非法结构消息可被拦截并返回错误，不直接导致主链路崩溃

## 下一步

1. 把 `engine.motion_plan` 从占位处理升级为可执行动作计划。
2. 增加协议回归测试（按消息类别分组）。
3. 收敛低价值兼容消息，减轻前后端协议负担。
