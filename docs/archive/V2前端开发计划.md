# V2 前端开发计划

快照日期：2026-04-21

## 目标

前端继续保持三层职责：

1. Live2D 渲染层
2. 动作引擎层
3. 桌宠交互 UI 层

当前策略：先稳定桌宠 UI 和通信，再推进真实动作引擎执行。

## 已完成

### 窗口与交互

- 桌宠窗口（pet）
- 覆盖层窗口（overlay）
- 系统设置窗口（settings）
- 历史窗口（history）
- 动作实验室窗口（action_lab）
- 右键菜单坐标链路已修复（按鼠标实际位置弹出）

### 对后端通信

- 已接入 V2 协议主链路（`input/output/control/system`）
- 支持文本发送、语音链路、播放完成回执、中断
- 用户侧单地址输入，内部派生 WS/HTTP
- `system.model_sync` 能同步模型与扫描结果

### 扫描结果可视化

- Action Lab 已可展示 `base_action_library`
- 支持筛选通道/域/关键词
- 支持选择动作原子、生成计划、发送 `engine.motion_plan` 测试消息

### 多模态补充

- 已支持“发送文本时附带实时桌面截图”的可开关能力

## 未完成

- 前端实时动作引擎执行路径（逐帧参数写入、时间轴、混合器）
- `engine.motion_plan` 的真实执行器（当前只发送测试消息，不驱动模型）
- 动作执行态与渲染态的统一诊断面板

## 下一阶段计划

1. 在前端实现最小执行器：支持基础参数通道与时长控制。
2. 把 Action Lab 生成计划直接接入执行器，完成“可见可播”闭环。
3. 补充动作执行回执（开始/完成/中断/失败）并接到 UI 状态。
4. 再做动作模板和组合策略优化，不提前过度设计。

## 完成标准（下一里程碑）

- 在桌宠窗口直接播放基于原子动作组合的测试计划。
- Action Lab 发送计划后，前端可看到实际动作变化而不是仅日志占位。
- 不依赖后端新动作系统，也能完成本地最小动作执行闭环。
