# V2 Live2D扫描设计草案

本文用于明确 AG99live V2 中 `live2ds` 扫描模块的职责边界、输入输出和第一阶段实现方向。

这部分设计的目标不是马上做完整字段定义，而是先回答三个最关键的问题：

1. 扫描器到底负责什么
2. 扫描器应该从模型目录里提取哪些信息
3. 扫描结果如何服务前端动作引擎，而不是重新走回 V1 的静态映射老路

当前这版设计再额外强调一个核心判断：

**扫描的第一重点是参数能力，expression 和 motion 主要作为补充能力、限制条件和 fallback 资源。**

## 1. 扫描器的定位

V2 的扫描器本质上不是“模型选择器”，也不是“动作规划器”，而是：

- 模型发现器
- 静态资源解析器
- 能力归纳器
- 运行时缓存生成器

一句话说清楚：

**扫描器负责把一个 Live2D 模型目录，转成“前端动作引擎可以消费的能力画像”。**

它不负责：

- 决定某一轮回复具体播哪个动作
- 生成逐帧动作结果
- 替代前端动作引擎
- 把 motion 语义彻底理解成完美标签

这些都应该是后续动作系统和前端 runtime 的职责。

## 2. 为什么 V2 需要独立扫描层

V1 里 Live2D 相关信息有几个问题：

- `model_info` 更像模型入口信息，不是完整能力描述
- 很多动作语义依赖人工维护的映射
- motion / expression / 参数能力没有分层
- 后端容易直接长成“替前端决定怎么动”的结构

V2 需要把这层拆开。

扫描层的价值在于：

- 让不同模型先统一落到一套“可比较的能力描述”
- 让前端动作引擎知道当前模型能做什么、不能做什么
- 让语义增强信息可以作为附加层存在，而不是强绑在原始模型文件上
- 让后续“重新扫描”成为标准流程，而不是手工整理 JSON

## 3. V2 扫描设计的总原则

### 3.0 参数优先，表情和动画后置

V2 扫描不应该再以“这个模型有哪些现成表情、有哪些现成动作”作为第一视角。

更合理的顺序应该是：

1. 先判断这个模型有哪些参数可被实时驱动
2. 再判断这些参数能覆盖哪些标准通道
3. 最后再看 expression 和 motion 能补什么、限制什么

原因很直接：

- V2 的核心目标是建立前端 runtime 动作引擎
- runtime 的主路径依赖参数驱动
- expression 更适合承担“基础情绪切换”或“特殊状态”
- motion 更适合承担“高质量 fallback”或“约束较强的整段表现”

所以扫描阶段要避免本末倒置：

- 不是先找一堆动作再想怎么套
- 而是先识别参数能力，再决定哪些场景需要 expression / motion 补位

### 3.1 两阶段产物

扫描结果必须分成两层：

- `事实层`
- `增强层`

事实层来自模型原始文件，强调“可重复扫描、尽量不带主观判断”。

例如：

- 模型主文件位置
- 表情文件列表
- motion 文件列表
- 参数列表
- 纹理列表
- 资源相对路径
- motion 时长、曲线、涉及参数

增强层来自可选附加文件或后处理，强调“可人工维护、可覆盖、可重新生成”。

例如：

- motion 语义标签
- 推荐场景
- 强度等级
- 排他关系
- expression 情绪归类
- 参数通道归一化映射

这样做的意义是：

**扫描器先还原事实，再做归纳，不把主观语义和原始资源搅在一起。**

### 3.2 先做静态扫描，不做运行时决策

扫描器只处理静态资源和静态分析结果。

不在扫描阶段做：

- “这一句回复应该选哪个动作”
- “这个动作现在应该播多久”
- “本轮语音应该切几段”

这些都属于动作引擎。

### 3.3 多模型兼容优先

扫描设计必须接受现实：

- 不同模型目录结构不一样
- 有的模型只有基础 `model3.json`
- 有的模型有 `motion_catalog.json`
- 有的模型有 VTube Studio 配置
- 有的模型动作很多但没有语义标签
- 有的模型参数丰富但没有适合直接复用的 motion

所以扫描层必须允许：

- 最低只靠原始 Live2D 文件工作
- 有附加文件时自动增强
- 缺某一类文件时平稳降级

## 4. 扫描输入来源

V2 第一阶段建议把扫描输入分成四类。

### 4.1 模型根目录

例如：

- `live2ds/Mk6_1.0/`

这是扫描的基本单位。

每个模型目录至少应该尝试发现：

- `*.model3.json`
- `*.moc3`
- 纹理目录
- `Expressions/`
- `Motions/`
- `*.physics3.json`
- `*.cdi3.json`

### 4.2 原始 Live2D 描述文件

主要包括：

- `model3.json`
- `exp3.json`
- `motion3.json`
- `cdi3.json`
- `physics3.json`

它们决定事实层的主要内容。

### 4.3 可选增强文件

第一阶段建议明确支持但不强依赖：

- `motion_catalog.json`
- `*.vtube.json`
- `items_pinned_to_model.json`

其中：

- `motion_catalog.json` 更像人工补充的 motion 语义目录
- `vtube.json` 可帮助识别参数用途、热键、追踪相关映射
- `items_pinned_to_model.json` 更偏业务扩展，不一定进入动作核心路径

### 4.4 扫描器自身默认规则

当模型没有增强文件时，扫描器仍应提供基础归纳。

例如：

- 从文件名推断 motion 基础标签
- 从 `model3.json` 的 motion group 推断 idle / talk / default
- 从参数 ID 推断头部、眼睛、嘴、身体等基础通道

这部分属于“规则型增强”，优先级低于人工增强文件。

## 5. 扫描输出应该是什么

V2 建议扫描器产出两类文件，而不是一个大而杂的 `model_info.json`。

### 5.1 全局模型索引

作用：

- 告诉系统当前发现了哪些模型
- 标明每个模型的基础入口
- 给前端和后端快速列出可选模型

它是“模型目录级索引”，不承担详细能力描述。

### 5.2 单模型能力画像

作用：

- 描述当前模型的完整能力
- 作为动作引擎和前端 runtime 的直接输入之一

建议里面至少包含以下几类信息：

- 模型基础入口信息
- 参数清单与参数分组
- expression 清单
- motion 清单
- motion group 信息
- 事实层解析结果
- 增强层语义结果
- 资源指纹和扫描时间

关键点不是字段细节，而是结构分层：

- `identity`
- `resources`
- `parameters`
- `expressions`
- `motions`
- `capabilities`
- `enhancements`
- `fingerprint`

## 6. 第一阶段必须扫描出的核心信息

### 6.1 模型基础身份

至少要能确定：

- 模型名
- 模型目录
- 主 `model3.json`
- 默认资源入口 URL 或相对路径

这部分解决“当前模型是谁”和“前端怎么加载它”。

### 6.2 参数能力

这部分是 V2 扫描的绝对主轴。

扫描器不能只知道“有多少个参数”，还要尽量归纳出参数类别。

第一阶段重点识别的高价值通道建议包括：

- 头部左右
- 头部上下
- 头部倾斜
- 身体左右
- 视线 X
- 视线 Y
- 左右眼开合
- 左右眼笑
- 嘴巴开合
- 嘴型/嘴角
- 呼吸

这里不要求一开始就精准覆盖所有参数，但至少要做：

- 原始参数 ID 收集
- 基础别名识别
- 通道归类
- 是否适合实时驱动的初步判断

这部分扫描结果未来会直接决定：

- 前端 runtime 能不能以参数驱动为主
- 哪些动作模板可以启用
- 哪些模型只能走弱化版动作系统

### 6.3 Expression 能力

扫描器要能提取：

- expression 名称
- 文件路径
- 影响了哪些参数
- 是不是更像“基础情绪表达”还是“特殊道具/状态切换”

这里非常关键，因为很多模型的 expression 并不都适合进入主动作系统。

例如：

- `Happy`
- `Angry`
- `Surprised`

适合作为基础情绪候选。

但像：

- `Tablet`
- `Pen`
- `Controller`

更像配件或模式切换，不一定应该纳入日常表达主路径。

所以扫描结果里最好区分：

- 基础表达型 expression
- 特殊状态型 expression

但这里要明确：

**expression 在 V2 里不是主动作通道，而是“参数主路径之外的补充层”。**

它主要承担三类职责：

- 当参数能力不足时，承担基础情绪表达
- 当某些状态更适合离散切换时，承担模式切换
- 当模型本身某种表情效果明显优于参数混合时，作为增强手段

所以扫描 expression 的重点不是“越多越好”，而是：

- 哪些 expression 适合常规对话主路径
- 哪些 expression 只适合特殊状态
- 哪些 expression 会与参数驱动冲突

### 6.4 Motion 能力

这部分不能只停留在“列文件名”。

第一阶段建议尽量扫描出：

- motion 文件路径
- motion 所属 group
- motion 时长
- 涉及参数
- 参数变化强度概览
- 是否更像 idle / talk / expressive / special

其中“时长”和“涉及参数”非常重要，因为这会直接影响后续动作引擎的 fallback 质量。

同时也要明确它在 V2 里的定位：

**motion 主要是限制条件和 fallback 资源，不应重新变成主路径。**

它更适合解决这些问题：

- 某个模型参数能力弱，无法实时做出足够好的表现
- 某类整段动作已有高质量现成资源，直接复用更稳
- 某些动作本身就是强约束的整段表演，不适合拆成参数通道

因此 motion 扫描更关注：

- 能不能安全调用
- 适合什么场景调用
- 调用时会覆盖哪些参数通道
- 会不会与当前 expression / 参数驱动冲突

### 6.5 资源完整性

扫描器还要判断模型是否可正常加载。

至少要能发现：

- 主文件缺失
- 引用资源不存在
- 纹理丢失
- expression / motion 引用断链

这部分很工程，但非常重要，因为它会直接决定“这个模型可不可用”。

## 7. Motion 扫描建议重点做什么

V2 的 motion 扫描建议单独作为一个重点子模块。

原因是：

- motion 是现成高质量资源
- 但 motion 语义最不统一
- 将来 fallback 很依赖它

第一阶段建议 motion 扫描至少产出三层结果。

### 7.1 文件层

最基础的信息：

- 文件名
- 相对路径
- 所属 group
- 是否被 `model3.json` 正式引用

### 7.2 结构层

从 `motion3.json` 中提取：

- 时长
- FPS
- 曲线数量
- 涉及参数 ID
- 是否有明显 Event

### 7.3 特征层

从参数曲线进一步归纳：

- 头部动作是否明显
- 身体动作是否明显
- 眼部变化是否明显
- 嘴部变化是否明显
- 整体幅度偏低还是偏高
- 更像平稳动作还是情绪动作

这层不是为了完美理解 motion，而是为了让前端 runtime 知道：

**这个 motion 是否适合拿来当 fallback，以及它大概属于哪一类。**

## 8. 参数归一化映射的设计思路

为了让前端动作引擎能够跨模型工作，扫描层需要把原始参数向“标准通道”靠拢。

建议不要直接把前端绑定死到某个模型参数名，而是建立“标准通道 -> 模型参数候选”的映射。

例如：

- `headYaw` -> `ParamAngleX`
- `headPitch` -> `ParamAngleY`
- `headRoll` -> `ParamAngleZ`
- `bodyYaw` -> `ParamBodyAngleX`
- `gazeX` -> `ParamEyeBallX`
- `gazeY` -> `ParamEyeBallY`
- `mouthOpen` -> `ParamMouthOpenY`
- `mouthSmile` -> `ParamMouthForm`

如果模型命名不标准，就用规则型匹配或增强文件补充。

这样前端动作引擎拿到的是：

- 这个模型支持哪些标准通道
- 每个标准通道映射到哪些真实参数
- 这些参数的安全范围和推荐强度

这一步会成为 V2 多模型通用性的核心。

## 9. VTube Studio 配置文件要不要扫

建议扫，但定位要谨慎。

`*.vtube.json` 不应成为唯一事实来源，因为它是某个外部工具的配置视角，不一定等同于模型原生能力。

但它很有价值，因为它常常包含：

- 参数输入映射
- 平滑设置
- 热键信息
- 某些 expression / motion 的使用习惯

因此 V2 建议把它作为“增强层参考”：

- 不覆盖原始事实层
- 只作为参数用途识别和行为建议来源

## 10. 推荐的扫描流水线

建议把扫描流程拆成六步。

### Step 1. 发现模型目录

遍历 `live2ds/` 下的一级目录，识别潜在模型根。

### Step 2. 建立原始资源清单

找到：

- `model3.json`
- 纹理
- expression
- motion
- 相关附属文件

### Step 3. 解析事实层

从原始文件中提取：

- 参数
- expression
- motion
- 文件引用
- 资源完整性

### Step 4. 构建增强层

合并：

- `motion_catalog.json`
- `vtube.json`
- 规则型命名推断

### Step 5. 归纳能力画像

输出：

- 标准通道映射
- 基础 expression 候选
- fallback motion 候选
- 模型支持等级

### Step 6. 写入缓存结果

生成：

- 全局索引
- 单模型能力画像
- 指纹信息

并支持下次启动快速复用。

## 11. 扫描结果如何服务动作引擎

扫描器的产物不应该直接长成“请播这个 motion”。

它更应该提供以下能力：

- 当前模型支持哪些标准通道
- 当前模型有哪些可作为基础情绪的 expression
- 当前模型有哪些可作为 fallback 的 motion
- 当前模型更适合参数驱动、expression 驱动，还是 motion 驱动

但优先级建议固定为：

1. 参数驱动主路径
2. expression 补充
3. motion fallback

也就是说，扫描器产出的是：

- `可用能力`
- `候选集合`
- `约束边界`

动作引擎再基于这些去做：

- 语义规划
- 时间轴规划
- 通道混合
- fallback 决策

## 12. 第一阶段建议的支持等级

为了避免一上来设计过重，建议扫描结果先把模型分成几个支持等级。

### Level A

参数通道丰富，expression 和 motion 都较完整。

适合：

- 参数驱动为主
- expression / motion 作为补充

### Level B

参数通道一般，但有较完整 expression 和 motion。

适合：

- expression + fallback motion 为主
- 参数驱动只做少量增强

### Level C

只有基础模型和少量资源。

适合：

- 极简参数驱动
- 基础 lip sync
- 有限 expression

这个等级不是给用户看的评级，而是给运行时决定策略的。

## 13. 第一阶段先不要做什么

先不要在扫描阶段做这些事情：

- 自动生成新的 `.motion3.json`
- 用大模型给所有 motion 自动写完整语义
- 做非常复杂的参数物理推断
- 直接在后端输出最终动作编排
- 把扫描结果设计成一次性不可更新的大 JSON

第一阶段只要做到：

- 稳定扫描
- 输出结构清晰
- 能支撑前端动作引擎做第一版 runtime

就已经足够有价值。

## 14. 推荐的 Phase 1 实现拆分

如果进入开发，建议按下面顺序落地：

1. 目录发现器
2. 原始模型解析器
3. expression 扫描器
4. motion 扫描器
5. 参数归一化器
6. 增强层合并器
7. 扫描缓存写入器

这样每一步都可单独验证，不会一开始就被大而全设计拖住。

## 15. 当前结论

当前可以把 V2 的 Live2D 扫描部分先定成下面这个方向：

- 扫描器是“能力画像生成器”，不是动作生成器
- 扫描优先级是“参数能力 > expression 补充 > motion fallback”
- 扫描结果必须分为事实层和增强层
- 第一阶段重点是参数、expression、motion 三大能力扫描
- `motion_catalog.json` 和 `vtube.json` 属于增强来源，不是唯一事实来源
- 扫描产物应该服务前端动作引擎，而不是重新变成 V1 的静态动作映射系统

一句话总结：

**V2 的 Live2D 扫描正确方向，不是“扫出一个模型地址”，而是“为动作 runtime 建立一份可用、可扩展、可降级的能力画像”。**

## 16. 基于 Mk6_1.0 的可行性验证

为了避免设计停留在抽象层，这里直接结合当前仓库里的 `Mk6_1.0` 样本做一次现实校验。

### 16.1 当前样本给出的信号

从当前模型目录可以确认：

- 参数总数约 `299`
- parameter group 约 `14`
- expression 数量约 `22`
- motion 数量约 `24`
- `model3.json` 中引用的 motion / expression 与实际文件是完整对齐的
- `motion_catalog.json` 已经完整覆盖这 `24` 个 motion

这说明至少对你当前主模型来说：

- 扫描器不是“想法”，而是有明确输入源的
- 第一阶段完全可以做出稳定扫描
- 不需要依赖大模型才能拿到基础能力画像

### 16.2 参数扫描的可行性很高

当前样本中，虽然参数总数接近 `300`，但并不是所有参数都同等重要。

粗看可以分成几类：

- 基础可控参数
- 物理参数
- expression / animation 参数
- 特殊业务参数

像当前样本里非常明显的高价值参数通道就已经存在：

- `ParamAngleX`
- `ParamAngleY`
- `ParamAngleZ`
- `ParamBodyAngleX`
- `ParamBodyAngleZ`
- `ParamEyeBallX`
- `ParamEyeBallY`
- `ParamEyeLOpen`
- `ParamEyeROpen`
- `ParamEyeLSmile`
- `ParamEyeRSmile`
- `ParamBrowForm`
- `ParamMouthForm`
- `ParamMouthOpenY`
- `ParamMouthX`
- `ParamBreath`

这说明第一阶段最核心的“标准通道映射”是完全有现实基础的。

换句话说：

**参数扫描这件事不但可行，而且很适合作为第一优先级。**

### 16.3 Expression 扫描可行，但更适合做分类而不是主驱动

当前样本里的 expression 文件结构也很规整，解析难度很低。

例如：

- `Neutral.exp3.json` 甚至是空参数集合
- `Happy.exp3.json` 主要修改的是 `Exp*` 类参数
- 某些 expression 会触发少量嘴部或开关类参数
- 少数 expression 还会涉及前臂开关这类特殊状态参数

这说明 expression 的自动扫描没有难度，但它的工程重点不是“能不能扫”，而是“扫出来后怎么分类”。

当前样本已经足够说明：

- 很多 expression 更像离散状态叠加层
- 不是所有 expression 都适合放进日常对话主路径
- expression 会与个别参数通道产生冲突，因此需要冲突标记

所以 expression 这块的可行做法不是“主驱动”，而是：

- 自动发现
- 自动分类
- 标记为基础情绪 / 特殊状态 / 可能冲突

### 16.4 Motion 扫描可行，但要承认“可解析”不等于“可完全理解”

当前样本里的 `motion3.json` 同样很规整：

- 有统一 `Meta`
- 有明确 `Duration`
- 有明确 `Fps`
- 有明确 `Curves`
- 每条 curve 都能拿到目标参数 ID

这意味着 motion 扫描在技术上完全可做：

- 时长可直接提取
- 涉及参数可直接提取
- 曲线数量可直接提取
- 可以进一步计算某些通道的变化强度

但这里也要保持克制：

**我们能自动解析 motion 结构，不代表我们能完全自动理解 motion 语义。**

例如：

- 我们能知道一个 motion 改了很多头部和眼睛参数
- 也能知道它强度偏高还是偏低
- 但不能只靠结构就稳定判断它是不是“委屈”“坏笑”还是“认真说明”

所以 motion 的最佳策略依然是：

- 结构信息自动提取
- 基础特征自动归纳
- 语义层优先吃 `motion_catalog.json`
- 没 catalog 时只做弱推断

### 16.5 `motion_catalog.json` 的价值非常明确

当前样本里 `motion_catalog.json` 已经完整覆盖全部 `24` 个 motion。

这是一件非常有价值的事，因为它说明：

- 人工增强层确实有必要
- 但它不需要替代原始扫描
- 它最适合承担 motion 的语义补全

所以现实上最稳的路线就是：

- 先从原始 `motion3.json` 提事实
- 再用 `motion_catalog.json` 做增强

这比让扫描器直接“猜完整语义”可靠得多。

## 17. 自动化边界判断

为了避免过度设计，这里把“能稳定自动化的部分”和“只能半自动的部分”分开。

### 17.1 能稳定自动化的部分

这些内容非常适合第一阶段直接落地：

- 模型目录发现
- `model3.json` 入口解析
- expression 文件发现
- motion 文件发现
- 资源完整性校验
- 参数清单提取
- 参数 group 提取
- 标准通道基础映射
- motion 时长提取
- motion 曲线参数覆盖提取
- expression 影响参数提取
- catalog / vtube 文件发现与载入

这些工作不需要大模型，也不需要复杂启发式，就能较稳定完成。

### 17.2 能半自动完成的部分

这些内容可以做，但要接受“结果带置信度”：

- 参数通道的进一步语义归类
- expression 的基础情绪 / 特殊状态分类
- motion 的 idle / talk / expressive / special 粗分类
- motion 强度等级推断
- 参数冲突关系初判

这部分适合做规则系统，不适合一开始就追求百分百正确。

### 17.3 不适合第一阶段自动化的部分

这些事情不是不能做，而是不应该放进 Phase 1：

- 自动为所有 motion 生成高质量完整语义说明
- 自动替所有模型建立完美参数语义图
- 自动生成最终动作编排
- 自动决定每轮对话用哪一个 motion
- 自动从 motion 反推复杂角色风格

这些要么很脆，要么容易把扫描器重新做成动作规划器。

## 18. 工程上的真实难点

当前看下来，V2 Live2D 扫描真正难的不是“读取文件”，而是“如何做有用的归纳而不越界”。

主要难点其实是下面四个。

### 18.1 参数很多，但真正有价值的只是一部分

当前样本中接近 `300` 个参数，但真正高价值、适合 runtime 主路径的核心参数只是其中一小部分。

所以扫描器不能只做“全量列表”，还必须做：

- 核心通道筛选
- 次级通道归档
- 物理参数降权
- 特殊业务参数隔离

### 18.2 expression 和 motion 会覆盖参数

当前样本已经说明：

- expression 不一定只改 `Exp*`
- 它也可能改嘴部参数、开关参数
- motion 会同时驱动大量核心参数和物理参数

因此扫描层必须产出“覆盖/冲突信息”，否则前端 runtime 在混合参数、expression、motion 时会很容易打架。

### 18.3 多模型一定不整齐

`Mk6_1.0` 现在比较规整，不代表所有模型都会这么规整。

所以扫描实现必须默认支持：

- 有 `catalog`
- 没 `catalog`
- 有 `vtube`
- 没 `vtube`
- 参数命名标准
- 参数命名混乱

这也是为什么第一阶段最好优先做“事实层稳定、增强层可选”的结构。

### 18.4 真正最难的是“扫描结果如何被前端用好”

扫描本身并不是整个系统最难的环节。

最难的是：

- 如何把扫描出的标准通道映射给前端 runtime
- 如何根据“支持哪些通道”切换动作模板
- 如何在参数、expression、motion 三层之间做调度

所以扫描模块一定要克制，只负责把地基打清楚。

## 19. 当前可行性结论

如果只问“这套扫描系统能不能做”，答案是：

**能做，而且第一阶段可行性很高。**

如果再细一点看：

- 参数扫描：高可行，应该优先落地
- expression 扫描：高可行，重点在分类和冲突标记
- motion 扫描：高可行，重点在结构提取和 fallback 画像，不在完整语义理解
- 增强层合并：高可行，尤其适合利用 `motion_catalog.json`

真正需要控制预期的是：

- 不要把“可扫描”误解成“可自动做完整动作规划”
- 不要把扫描器设计成替前端引擎做决策
- 不要在 Phase 1 追求所有模型的完美理解

## 20. 推荐的落地判断

如果我们现在开始实现，最合理的结论是：

1. 扫描模块先做，而且值得做
2. 第一版就以参数主轴来设计
3. expression / motion 先做成补充能力与限制信息
4. catalog / vtube 一律作为增强层
5. 前端 runtime 后续直接消费“标准通道 + 冲突边界 + fallback 候选”

一句话总结这次可行性判断：

**这件事不是“能不能做”的问题，而是“很适合做，但一定要把参数主轴和扫描边界守住”。**
