# 04 分阶段执行计划

## 总原则

这是一次破坏性重构。

目标不是长期兼容固定 12 轴，而是尽快把核心抽象迁移到动态主轴。

原则：

- 每个阶段独立提交。
- 每个阶段都有可验证结果。
- 不用兼容逻辑掩盖错误。
- 旧实现由 Git 保留，不在新架构里长期背负。

严格开发模式：

- 不使用静默 fallback 掩盖错误。
- 不自动吞掉非法 axis、非法 binding。
- 数值越界不是结构错误：应夹到对应 min/max 极限值，并写入 warning/diagnostics。
- 主参数错误允许按阈值继续：默认错误率不超过 30% 时忽略错误项并继续；超过阈值才 hard fail。
- 允许 warning 的场景必须进入 diagnostics，并能在开发 UI 或日志中定位。
- 保存、编译、runtime 写参失败必须具备明确错误码或诊断字段。
- 兼容旧数据只能作为显式迁移步骤，不能作为长期双轨逻辑隐藏在运行时。

## Phase 1：新增 SemanticAxisProfile 类型和默认 profile

目标：让代码拥有动态主轴的数据结构。

任务：

- 新增 profile 类型定义。
- 新增默认 profile，使用当前固定 12 轴作为初始模板。
- 标记控制权：
  - primary: head/gaze/eye expression/mouth_smile
  - derived: body_yaw/body_roll/brow_bias
  - runtime: mouth_open
- 不接入 compiler，只先建立数据结构。

验收：

- TypeScript 类型通过。
- 默认 profile 可被加载和展示。

## Phase 2：后端 Profile 存储、扫描回填与版本字段

目标：让后端成为 profile 的事实来源，并在模型目录中落盘。

任务：

- 在 `live2ds/<model_name>/ag99/semantic_axis_profile.json` 持久化 profile。
- 基于现有 model sync / scan 数据生成初版 `semantic_axis_profile`。
- 固定模板外的扫描参数先以 `debug` 轴进入 profile，供后续人工提升为 primary / derived / runtime。
- 计算 Live2D 文件夹 hash。
- 增加 `profile_id`、`revision`、`source_hash`、`user_modified` 等基础字段。
- hash 未变化时复用 profile。
- hash 变化时标记 profile stale，不静默覆盖用户修改。

验收：

- 当前 Mk6 模型可以生成或回填 profile。
- profile 以 JSON 持久化在模型目录。
- 扫描失败时有显式 error/warning。

## Phase 3：前后端 Profile 读写协议

目标：建立前端读取、编辑、保存后端 canonical profile 的最小协议。

任务：

- 新增 profile 同步协议或同步 payload。
- 前端读取当前模型 canonical profile。
- 前端保存编辑结果时显式提交给后端。
- 前后端使用同一组 `profile_id + revision` 做版本识别。
- profile 不一致时显式报 warning/error，不静默覆盖。

验收：

- 前端能读取当前模型 profile。
- 前端能把修改保存回后端模型目录。
- 后端和前端日志中能看到同一个 `profile_id` 和 `revision`。

## Phase 4：前端 Profile Editor / Action Lab 接入

目标：允许用户选择主轴和联动关系。

任务：

- 新增独立 Profile Editor 辅助窗口，不再放在 Action Lab 内。
- 动态展示 profile 中所有 scanned axes。
- 用户可修改：
  - control_role
  - label/description
  - positive/negative semantics
  - neutral/range
  - parameter binding
  - coupling 开关和强度
- 保存 profile 到后端。

验收：

- Profile Editor 不再依赖固定 12 轴生成主轴编辑 UI。
- Profile Editor 支持用户维护参数排除关键词，默认隐藏 hair/bind/physics/Phy 这类物理或绑定参数，避免扫描参数过多污染主轴选择。
- SettingsWindow、MotionTuningPanel、BaseActionPreviewPanel 和现有调参历史必须走 v2 profile-driven 数据源；v1 playback 不再作为兼容路径保留。
- 用户可以把某个参数设为 primary/derived/runtime/debug。
- 用户可以配置 `head_yaw -> body_yaw` 这类联动。
- 自定义主轴必须补齐必填字段并显式确认配置后才能保存。
- 后端拒绝非法 axis id、非有限数字、反向 range、越界 neutral/range、跨轴重复 binding、负 `scale/deadzone/max_delta`、非法 coupling 和 coupling cycle。

## Phase 4.5：SemanticAxisProfile 保存协议硬化

目标：在进入 `motion_intent.v2` 前，先让 profile 保存协议具备明确 ack/fail，不再依赖 `system.model_sync` 回刷作为保存成功确认。

任务：

- 新增 `system.semantic_axis_profile_saved`。
- 新增 `system.semantic_axis_profile_save_failed`。
- save 请求必须带 `request_id`、`model_id` 或 `model_name`、`profile_id`、`expected_revision`。
- saved 响应必须带 `request_id`、`model_id` 或 `model_name`、`profile_id`、`revision`、`source_hash`。
- failed 响应必须带 `request_id`、`model_id` 或 `model_name`、`profile_id`、`error_code`、`message`。
- 前端保存状态只接受同 `request_id` 的 ack/fail，不再从通用 `control.error` 或 `system.model_sync` 推断保存结果。
- `system.model_sync` 仍可作为 profile 刷新来源，但不再承担保存确认职责。
- 旧的 12 轴兼容路线文档必须标记为 superseded 或移动到 `docs/archive`。

验收：

- 已完成。保存成功有专用 `system.semantic_axis_profile_saved`。
- 已完成。保存失败有专用 `system.semantic_axis_profile_save_failed`。
- 已完成。revision 冲突、source_hash/stale 冲突、schema 校验失败会返回明确 `error_code/message`。
- 已完成。Profile Editor 只按 `request_id` 匹配保存结果，unrelated `control.error` 不再作为保存结果。
- 已完成。save 请求要求 `request_id/model_name/profile_id/expected_revision/profile` 必填。

## Phase 6：Prompt 生成改为 profile-driven

目标：后端 prompt 不再硬编码固定 12 轴。

任务：

- prompt 从 profile 中生成 primary/hint 轴说明。
- runtime/derived/ambient 轴不进入主输出列表。
- 输出 schema 切到 `engine.motion_intent.v2`。

验收：

- 已完成。realtime prompt 中不再出现固定 12 轴说明。
- 已完成。prompt 只暴露 profile 中的 `primary/hint` axes。
- 已完成。LLM 输出 `engine.motion_intent.v2` 动态 axes。
- 已完成。forbidden/unknown axis 出现时解析层 reject。
- 已完成。stale/invalid profile 会阻止 realtime/inline motion contract 生成。

## Phase 7：ParameterPlan v2 Runtime

目标：Runtime 支持动态参数写入。

任务：

- 新增 `engine.parameter_plan.v2`。
- `parameters[]` 直接声明 `parameter_id/target_value/weight`。
- Runtime 只根据 parameter_id 写入，不再理解固定轴名。
- 第一版严格处理：任一 `parameter_id` 不存在、不可写、binding 不可解析、重复 parameter 或非法 number/weight/source 时，整条 plan fail。
- `target_value` 越界时不 fail，夹到当前 Live2D 参数 min/max 极限值并记录 warning。
- 失败必须输出 parameter 级 diagnostics，不允许静默跳过。

验收：

- 已完成核心代码。Live2D runtime 已接受 `engine.parameter_plan.v2`。
- 已完成。v2 runtime 只按 `parameters[].parameter_id` resolve/write，不读取 `key_axes`。
- 已完成。preview player 可解析并下发 v2 plan。
- 已完成。缺失 parameter、重复 parameter、不可写参数、非法 number、非法 weight 会拒绝整条 plan，并通过 `getDirectParameterPlanError()` 暴露原因。
- 已完成。越界 target 会夹到模型参数 min/max 并输出 warning，不再拒绝整条 plan。
- 待实机验证。代码与构建已通过，仍需要在真实 Live2D 模型上观察动作效果和参数范围。

## 验收总标准

重构完成时必须满足：

- 不再把固定 12 轴作为架构核心。
- 主控参数由 profile 决定。
- prompt 根据 profile 生成。
- 前端可编辑主轴和联动关系。
- ModelEngine 可编译动态 axes。
- Runtime 可执行动态 parameter plan。
- 所有派生/runtime/ambient 控制权可诊断。
