# 01 动态主轴总体方案

## 核心变化

从固定轴：

```text
DIRECT_PARAMETER_AXIS_NAMES = [固定 12 轴]
```

改为动态 profile：

```text
SemanticAxisProfile.axes[]
```

未来代码不应依赖固定 12 轴判断“哪些是主参数”。

## 新链路

```text
Live2D model files
-> scanner
-> semantic_axis_profile.json
-> profile distribution
-> prompt generator
-> LLM outputs engine.motion_intent.v2
-> ModelEngine compiles with profile
-> engine.parameter_plan.v2
-> Live2D runtime writes concrete parameters
```

其中 `profile distribution` 是必须显式存在的一层：

- 前端 ModelEngine 需要当前 profile。
- Action Lab / Profile Editor 需要当前 profile。
- 后端 prompt generator 也需要当前 profile。

如果没有统一分发机制，后端和前端会各自持有不同版本的 profile，最终导致 prompt、编译器和 runtime 理解不一致。

## 概念定义

### Semantic Axis

语义轴不是 Live2D 原始参数，而是 AG99live 定义的可控语义槽位。

示例：

```text
head_yaw
mouth_smile
eye_expression
body_follow_yaw
ear_react
tail_sway
```

一个 semantic axis 可以绑定一个或多个 Live2D 参数。

### Parameter Binding

将 semantic axis 映射到具体 Live2D 参数。

示例：

```json
{
  "axis_id": "head_yaw",
  "parameter_id": "ParamAngleX",
  "input_range": [0, 100],
  "output_range": [-30, 30]
}
```

### Control Role

每个轴必须声明控制权：

| role | 含义 |
|---|---|
| `primary` | LLM 主控，进入 prompt |
| `hint` | LLM 可弱控制，但 ModelEngine 可重算 |
| `derived` | ModelEngine 根据其他轴派生，不进入 prompt 主列表 |
| `runtime` | 由音频/口型/眨眼等 runtime 控制 |
| `ambient` | 待机/呼吸/背景动作控制 |
| `debug` | 只用于调试和 Action Lab |

## 动态主轴后的协议方向

### motion_intent.v2

```json
{
  "schema_version": "engine.motion_intent.v2",
  "profile_id": "Mk6_1.0.semantic.v1",
  "mode": "expressive",
  "emotion_label": "happy",
  "duration_hint_ms": 1200,
  "axes": {
    "head_yaw": { "value": 54 },
    "head_roll": { "value": 46 },
    "mouth_smile": { "value": 72 },
    "eye_expression": { "value": 58 }
  },
  "summary": {
    "axis_count": 4
  }
}
```

规则：

- LLM 只输出 profile 中 `control_role=primary` 的轴。
- `hint` 轴可以输出，但 prompt 中必须标注“非必要”。
- `derived/runtime/ambient/debug` 不应由 LLM 主动输出。
- 缺失 primary axis 时由 ModelEngine 按 neutral/default 补齐，并输出 warning diagnostics。

### parameter_plan.v2

```json
{
  "schema_version": "engine.parameter_plan.v2",
  "profile_id": "Mk6_1.0.semantic.v1",
  "mode": "expressive",
  "emotion_label": "happy",
  "timing": {
    "duration_ms": 1200,
    "blend_in_ms": 180,
    "hold_ms": 720,
    "blend_out_ms": 300
  },
  "parameters": [
    {
      "axis_id": "head_yaw",
      "parameter_id": "ParamAngleX",
      "target_value": 0.2,
      "weight": 1
    }
  ],
  "diagnostics": {
    "primary_axes": ["head_yaw", "mouth_smile"],
    "derived_axes": ["body_follow_yaw"],
    "runtime_axes": ["mouth_open"]
  }
}
```

规则：

- Runtime 不再要求固定 12 轴。
- Runtime 只写 `parameters[]` 中声明的具体 `parameter_id`。
- 每条 parameter 都必须能追溯到 `axis_id`。
- `target_value` 必须明确是“参数原生空间的目标值”，不是 0..100 的语义值。
- `target_value` 的合法范围由 `parameter_bindings.output_range` 或 runtime 绑定解析决定。

也就是说：

- `motion_intent.v2.axes[*].value` 是语义值，通常在 `0..100`。
- `parameter_plan.v2.parameters[*].target_value` 是最终写给 Live2D 参数的目标值，处于参数原生空间。

这两个层级不能混用。

## 旧 12 轴如何处理

因为当前处于开发阶段，建议不做长期兼容层。

迁移策略：

- 旧 12 轴可作为默认 profile 的初始模板。
- 新代码不再新增对 `DIRECT_PARAMETER_AXIS_NAMES` 的依赖。
- 新 UI 以 profile 为数据源。
- 旧 v1 协议在重构完成后删除，而不是长期双轨维护。

## 设计风险

| 风险 | 处理方式 |
|---|---|
| 自动扫描不准 | 前端 profile editor 人工校正 |
| profile 太复杂 | 第一版只暴露 primary/derived/runtime 和基本范围 |
| prompt 太长 | 只把 primary/hint 轴写入 prompt |
| runtime 写参失败 | parameter_plan.v2 必须强诊断，不允许静默跳过 |
| 调参历史失效 | 当前阶段接受破坏性重构，由 Git 保留旧实现 |
