# 后端主导数据边界与执行计划

快照日期：2026-04-29

## 一句话结论

动态主轴执行链已经闭环；下一步不先重写 `ModelEngine`，而是先把长期数据的归属彻底收回后端。

新的固定原则是：

```text
后端负责长期事实
前端负责临时编辑、实时执行和调试展示
```

## 背景

当前 v2 主链路已经完成：

```text
semantic_axis_profile
-> profile-driven prompt
-> engine.motion_intent.v2
-> frontend ModelEngine
-> engine.parameter_plan.v2
-> Live2D runtime
```

这条链路已经不再依赖固定 12 轴 v1 执行协议，说明“动作语义 -> 参数执行”的主边界已经成立。

当前真正混乱的地方，不再是执行链本身，而是“哪些数据应该长期保留、由谁保管”还没有完全收口：

- `Profile Editor` 已独立，但 `Action Lab` 仍然反向依赖前端编辑副本。
- 动作调参样本正式事实源已切到后端；前端只保留跨窗运行时投影，不再通过 `snapshot/localStorage` 持久化样本副本。
- `DesktopRuntimeSnapshot` 同时混合运行态、工具态和部分长期数据。
- `PetDesktopView` 既像运行时主控层，又像工具数据中台。

## 新的架构共识

### 1. 后端拥有长期事实

以下数据以后端为准：

- `semantic_axis_profile`
- 已确认的动作调参参考样本
- few-shot 参考样本池
- 以后如果独立出的动作定义/动作库
- 模型元数据、历史会话、资源扫描结果

这些数据的共同特点：

- 需要跨重启保留
- 需要作为运行时事实源
- 不能依赖某个前端窗口是否存在

### 2. 前端只保留临时草稿和实时状态

以下数据以前端为主，但不是事实源：

- 正在编辑但还没保存的 `Profile` 草稿
- 正在微调但还没确认的样本草稿
- 当前播放队列、待播动作、音频同步状态
- 窗口布局、筛选条件、面板开关、选中项
- 最近几次真实播放记录，用于本地临时回放和调试

这些数据的共同特点：

- 生命周期短
- 主要服务交互和调试体验
- 可以在重开窗口后丢失或重新初始化

### 3. “确认保存”是长期数据进入后端的边界

长期数据不应该被前端静默持久化。

推荐流程：

```text
后端 canonical 数据
-> 前端生成临时草稿
-> 用户编辑 / 预览 / 校验
-> 用户显式点击保存
-> 后端写入并确认
-> 前端刷新为最新 canonical 投影
```

这个原则同时适用于：

- `Profile`
- few-shot 参考样本
- 未来可能独立出的动作定义

## 数据归属表

| 数据 | 长期归属 | 前端角色 | 备注 |
| --- | --- | --- | --- |
| `semantic_axis_profile` | 后端 | 编辑草稿、展示、保存入口 | 后端 `live2ds/<model>/ag99/semantic_axis_profile.json` 是 canonical source |
| Profile 编辑中的未保存修改 | 前端 | 本地临时草稿 | 不能反向作为运行时事实源 |
| 最近真实播放记录 | 前端 | 调试 / 重放 / 样本起点 | 可丢弃，不是长期事实 |
| 已确认动作调参参考样本 | 后端 | 创建、编辑、启停 | 保存后可进入 few-shot 参考池 |
| few-shot 参考样本池 | 后端 | 查看投影 | 用于 selector prompt，不属于前端事实 |
| `turn` 播放状态、音频状态、待播动作 | 前端 | 实时调度 | 属于 `ModelEngine` / runtime |
| 辅窗 snapshot | 前端 | 跨窗投影 | 只传投影，不承载长期事实 |

## 模块职责收口

### 后端职责

- 维护 `semantic_axis_profile` 的 canonical store
- 提供 profile 读取、保存、冲突保护和保存确认协议
- 维护动作调参参考样本的持久化存储
- 组装 few-shot 参考样本，供 realtime selector prompt 使用
- 继续负责动作语义生成，不负责编译执行计划

### 前端职责

- `Profile Editor`：编辑 profile 草稿，做前端校验，显式提交保存
- `Action Lab`：预览、回放、微调、创建候选样本
- `ModelEngine`：接收动作意图，做本地编译、时序编排和播放
- `DesktopBridge`：只做跨窗投影和命令通道，不做长期事实源

### 运行时主窗口职责

`PetDesktopView` 后续应收缩为运行时主控层，只保留：

- 后端连接
- 当前模型同步
- `ModelEngine` 调度
- Live2D 播放执行
- 对辅窗发布只读运行态投影

不再长期承担：

- 正式参考样本存储
- Profile 编辑副本事实源
- 工具域的数据中台职责

## 当前代码中的主要偏差

### 1. `Action Lab` 仍依赖前端 Profile 编辑副本

这与“后端 canonical profile 才是事实源”的目标冲突。

目标改法：

- `Action Lab` 只读取 runtime 同步下来的当前 canonical profile 投影
- `Profile Editor` 只维护自己的临时草稿
- 两者通过“保存到后端后的刷新”间接一致，不直接共用编辑副本

### 2. 动作调参样本仍主要保存在前端

当前 few-shot 参考样本的正式生命周期还不在后端。

目标改法：

- 前端保留最近播放记录和未确认草稿
- 用户确认后，提交后端保存为正式参考样本
- 后端决定哪些样本进入 selector few-shot 参考池

### 3. `DesktopRuntimeSnapshot` 混入了工具域和长期数据

当前 snapshot 同时承担：

- 运行态同步
- 工具窗共享
- 本地持久化
- 部分长期数据缓存

目标改法：

- runtime snapshot 只保留运行态和必要 UI 投影
- tool snapshot 独立，且只承载临时工具态
- 长期数据改为后端读取和保存

### 4. `PetDesktopView` 职责过载

当前它同时承担：

- 运行时编排
- 工具数据聚合
- 跨窗单写者
- 保存代理

目标改法：

- 保留运行时编排和执行
- 工具域数据迁出
- 保存链路尽量收敛到明确的服务边界

## 下一步执行计划

### 第一批：Profile 收口到“后端真源 + 前端草稿”

目标：

- `Profile Editor` 只编辑本地草稿
- `Action Lab` 不再读取 profile 编辑副本
- runtime 只消费后端同步下来的 canonical profile 投影

任务：

- 梳理 `profileAuthoringSnapshot` 和 runtime profile 投影的使用点
- 让 `Action Lab` / `MotionTuningPanel` 只依赖 canonical profile 投影
- 保存成功后重置或刷新前端草稿
- 去掉前端长期保留 profile 编辑副本作为事实源的路径

验收：

- `Action Lab` 打开时不依赖未保存草稿
- 关闭 `Profile Editor` 后，runtime 和 `Action Lab` 行为不受影响
- 保存 profile 后，前后端 revision/source_hash 仍一致

### 第二批：动作调参样本和 few-shot 参考样本后端化

目标：

- 前端只保留临时调参记录和未确认草稿
- 正式参考样本保存在后端

任务：

- 设计后端样本存储结构
- 增加样本的读取、保存、删除、启停协议
- 前端把“保存样本”改为“确认后提交到后端”
- few-shot prompt 只读取后端正式参考样本

验收：

- 清空前端本地缓存后，正式参考样本仍可恢复
- 启用/停用参考样本不依赖 `localStorage`
- selector few-shot 输入来自后端正式样本池

### 第三批：拆分 `Action Lab` 工具域

目标：

- `Action Lab` 不再依附运行时 snapshot 存活
- 运行时和工具域的状态边界清晰

任务：

- 把预览工具状态、历史调参状态从 runtime snapshot 拆出
- 收缩 `PetDesktopView`
- 让 `Action Lab` 成为标准辅窗工具，而不是 runtime 的隐含扩展

验收：

- `DesktopRuntimeSnapshot` 不再承载正式参考样本和 profile 编辑副本
- `PetDesktopView` 的职责缩回运行时主控层

### 第四批：协议治理与测试补齐

目标：

- 前后端对 `engine.motion_*` 和新数据协议的解释一致
- 把新边界变成测试基线

任务：

- 统一 `engine.motion_*` 的 schema 解释
- 修掉当前 `input_value = null` 等协议漂移
- 增加 profile / sample / runtime snapshot 边界测试
- 增加前端 ModelEngine / ActionLab / bridge 关键测试

验收：

- 前后端对同一 payload 的合法性结论一致
- 新增后端样本链路具备自动化测试
- 前端工具域拆分具备类型和构建验证

## 最终目标

完成本轮收口后，应形成稳定结构：

```text
后端
  Profile 真源
  参考样本真源
  few-shot 参考池
  动作语义生成

前端
  Profile 临时编辑器
  Action Lab 调试台
  ModelEngine 实时执行器
  Live2D 展示与播放
```

在这个结构下：

- `Profile` 是模型长期能力定义
- few-shot 参考样本是后端动作生成输入材料
- `ModelEngine` 只负责动作编排和执行
- 前端是实时工作台，不再承担长期事实保管
