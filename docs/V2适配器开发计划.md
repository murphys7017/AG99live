# V2 适配器开发计划

本文用于记录 AG99live V2 阶段中 `adapter` 侧的开发计划。

这份文档的目标不是定义协议字段细节，也不是直接设计最终实现，而是先把适配器在 V2 中的职责、边界、优先级和开发顺序写清楚，避免后续开发时又回到 V1 那种“后端承担过多前端 runtime 逻辑”的模式。

## 1. 文档目的

本文主要回答以下问题：

1. V2 的适配器到底负责什么
2. V2 的适配器明确不负责什么
3. 适配器应该按什么顺序开发
4. 为什么当前阶段适配器不应抢在前端 runtime 之前过度展开

## 2. V2 适配器的定位

V2 中的适配器不是“动作总控中心”，而是：

- AstrBot 侧的协议桥接层
- 本地模型资源的管理与提供层
- 低频分析与基础产物生成层

一句话总结：

**V2 适配器负责把模型和基础能力准备好，并将 AstrBot 对话链路接到桌面端 runtime；它不负责替前端完成实时动作生成。**

## 3. V2 适配器负责的内容

### 3.1 AstrBot 协议桥接

这是适配器最基础、也必须保留的职责。

包括但不限于：

- AstrBot 对话消息的收发桥接
- 文本输入输出适配
- 图片输入输出适配
- 语音输入输出适配
- STT / TTS / 对话链路的接入
- 与桌面端 runtime 的消息协议适配

### 3.2 模型资源管理

适配器负责管理 Live2D 模型相关资源，例如：

- 模型目录管理
- 当前启用模型的记录
- 模型资源路径解析
- 模型静态资源提供
- 多模型切换时的资源侧准备

这里的重点不是“让模型动起来”，而是“保证模型能被正确发现、读取、分发和加载”。

### 3.3 模型扫描

适配器负责对当前模型进行基础扫描，扫描内容包括但不限于：

- `model3.json`
- `cdi3.json`
- `physics3.json`
- `exp3.json`
- `motion3.json`

扫描的目标不是直接做动作编排，而是产出一份基础能力认知，例如：

- 模型有哪些资源
- 有哪些表情
- 有哪些动作
- 有哪些参数
- 哪些参数可能适合后续前端 runtime 使用

### 3.4 基础配置或 manifest 生成

适配器需要基于扫描结果，生成一份供前端 runtime 使用的基础配置产物。

当前阶段不定义字段细节，但可以先明确其职责：

- 描述当前模型的基本能力
- 描述模型资源概况
- 为前端 runtime 提供“模型能力说明书”
- 为前端动作系统提供初始输入

这个产物的本质不是“最终动作方案”，而是“前端动作引擎的基础输入”。

### 3.5 配置缓存与自动生成

当前已经达成的方向是：

- 如果基础配置不存在，则自动生成
- 如果需要更新，则以后应通过“自动检测变化并重建”或“明确的刷新入口”处理
- 当前阶段允许先保持简单策略，但长期不应依赖人工长期手动维护大量配置文件

可以先接受的工程策略是：

- 没有就生成
- 有问题时允许删除旧产物再重建

但长期目标仍然应是：

- 自动检测失效
- 自动重建
- 保持最小人工维护成本

## 4. V2 适配器不负责的内容

下面这些内容在 V2 中应明确属于前端 runtime，而不是适配器：

- 实时动作编排
- 参数逐帧计算
- expression 与 motion 的最终播放时机决策
- 音频时长驱动的动作切段
- 强调点插入
- 实时参数曲线生成
- UI 状态机
- 桌面交互细节

尤其需要反复提醒自己：

**适配器不应该重新长成一个“替前端做动作决策”的系统。**

如果后续某个设计需要让适配器开始回答：

- 这句回复该怎么动
- 当前第 2.3 秒应该切什么动作
- 哪个眉毛参数现在应该拉多少

那就说明职责边界已经偏了。

## 5. 为什么适配器不是当前第一优先级

V2 当前阶段最难、最不确定、也最需要先验证的部分，其实在前端 runtime，而不在适配器。

原因包括：

- 动作系统的新路径是参数化实时驱动，而不是沿用 V1 静态动作映射
- 目前最需要先证明的是前端 runtime 能否把“模板 + 参数 + 时长 + 节奏”跑起来
- 如果前端 runtime 的主链路没有定下来，适配器过早细化会导致返工
- 前期完全可以用固定模型和 mock 数据验证前端方案

因此，当前阶段的研发顺序应当是：

1. 前端 runtime 先验证
2. 适配器再按前端真实需求补齐

这并不意味着适配器不重要，而是意味着：

**适配器应在正确的时间展开，而不是抢在动作系统核心路径尚未稳定之前过度设计。**

## 6. V2 适配器的阶段计划

### 阶段 A：边界明确

这一阶段的目标不是写很多代码，而是先把适配器边界定住。

需要明确：

- 适配器与前端 runtime 的职责边界
- 模型资源由谁管理
- 扫描结果大致承担什么作用
- 哪些内容属于基础产物，哪些属于实时运行时逻辑

这一阶段的产出主要是文档和边界判断。

### 阶段 B：最小资源与扫描能力

当前端 runtime 已经开始验证参数化动作系统后，适配器可进入最小实现阶段。

优先目标：

- 模型目录管理
- 当前模型读取
- 静态资源提供
- 最基础的模型扫描
- 基础配置自动生成

这一阶段不追求复杂规则，只追求让前端 runtime 不再完全依赖手写 mock 数据。

### 阶段 C：配置缓存与刷新机制

当前端对基础产物有更明确需求后，再补：

- 缓存机制
- 自动重建逻辑
- 强制刷新入口
- 多模型切换时的更新策略

### 阶段 D：协议稳定化

等前端 runtime 与适配器之间的信息交换逐渐稳定后，再对协议进行收敛。

这一阶段才适合进一步整理：

- 配置下发逻辑
- 版本兼容逻辑
- 资源与运行时的同步策略

## 7. 当前阶段建议的开发态度

V2 适配器开发当前最重要的原则是：

- 不抢前端 runtime 的职责
- 不急着定义过细字段
- 不急着做完整扫描体系
- 先保留足够简单、但方向正确的结构

当前阶段更适合把适配器看作：

- 后续必需的基础设施
- 但不是最先需要打透的核心风险点

## 8. 当前结论

当前已确定的结论如下：

- V2 适配器继续保留，并且仍然是系统重要组成部分
- 但其定位应收缩为“协议桥接 + 模型资源管理 + 低频分析 + 基础配置生成”
- V2 适配器不负责实时动作引擎逻辑
- 模型扫描与基础配置生成应保留在适配器侧
- 前端 runtime 的验证优先级高于适配器细化
- 当前阶段适配器文档先立边界，不急于下沉字段结构

一句话总结：

**V2 适配器是前端 runtime 的支撑层，而不是动作系统本体。**
