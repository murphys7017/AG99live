# 动作分析并入单次请求可行性评估

快照日期：2026-04-22
范围：在 AstrBot Provider 模式下，将“回复文本 + 动作计划”合并为一次主请求完成

## 目标定义

目标不是“让前端直接调模型”，而是：
- 一次主 LLM 请求同时产出可展示文本与动作语义。
- adapter 在同一轮内解析动作计划并下发 `engine.motion_plan`。
- 避免当前“第二次 LLM 调用”带来的额外延迟与串轮风险。

## 当前链路（现状）

1. 主对话请求产出 `output.text`。
2. `turn_coordinator` 触发异步二次调用生成动作计划：`adapter/adapter/turn_coordinator.py:186,458-509`。
3. `realtime_motion_plan` 再调用 provider：`adapter/adapter/realtime_motion_plan.py:68-87`。
4. 下发 `engine.motion_plan`。

现状问题：
- 固定多一次模型调用，延迟上限受 `realtime_motion_timeout_seconds`（默认 8s）约束（`adapter/_conf_schema.json:86-90`）。
- 存在 stale-turn 串轮风险（详见审阅报告）。

## 可行性结论

结论：**高可行，且建议尽快改为单次请求主链路。**

理由：
- 协议和前端已经有 `engine.motion_plan` 通道，可复用。
- 历史桥接已具备内联动画标签清洗能力：`<@anim {...}>` 会被去除（`adapter/adapter/history_bridge.py:12,355-359`）。
- AstrBot 文档支持在 LLM 请求前后钩子改写内容，也支持统一 `llm_generate` 能力（见文末参考）。

## 方案对比

### 方案 A（推荐）- 主回复内嵌结构化动作标签

做法：
- 让主模型在回复尾部输出结构化块，例如：`<@anim { ...json... }>`。
- `turn_coordinator.emit_message_chain` 解析并剥离标签：
1. 纯文本部分走 `output.text`。
2. 结构化部分转为 `engine.motion_plan`（同 `turn_id`）。

优点：
- 只需一次 LLM 调用。
- 与当前消息流改动最小，不依赖新增前端协议。
- 可与现有 `history_bridge` 清洗逻辑天然兼容。

风险与控制：
- 模型输出 JSON 可能不规范 -> 解析失败时回退“仅文本，无动作”。
- 用户提示词注入伪标签 -> 仅解析“assistant 输出末尾第一段合法标签”。

### 方案 B - 使用 AstrBot `on_llm_response` 钩子抽取结构化动作

做法：
- 在 `on_llm_response` 钩子里统一解析和改写响应。

优点：
- 逻辑集中，理论上可多插件复用。

缺点：
- 对当前 adapter 内部职责侵入更高，调试路径更绕。
- 仍需在 adapter 侧关联 turn_id 和转发策略。

### 方案 C - 保持双调用（当前）

优点：
- 改动最小。

缺点：
- 延迟和串轮风险持续存在。
- 成本更高，链路更复杂。

## 推荐落地路径（低风险分阶段）

### 阶段 1：并行影子验证（不影响现网）

- 主回复中先生成 `<@anim {...}>`，但仅解析记录，不立刻播。
- 保留当前二次调用链路，用于对比一致性。

验收：
- 结构化块解析成功率 >= 95%。
- 解析失败时文本链路 100% 不受影响。

### 阶段 2：切主链路

- 关闭默认二次调用（`enable_realtime_motion_plan=false` 或代码短路）。
- 主链路解析成功则直接下发 `engine.motion_plan`。
- 失败回退“仅文本”。

验收：
- 平均回复到动作首包延迟显著下降。
- 不再出现跨轮次动作错配。

### 阶段 3：清理与收敛

- 删除或降级旧二次调用路径为 debug/兜底。
- 补齐文档和回归测试。

## 关键实现点（文件级）

1. `adapter/adapter/turn_coordinator.py`
- 在 `emit_message_chain()` 内增加“文本/动画标签拆分”。
- 广播 `engine.motion_plan` 时强制使用当前轮 `turn_id`（或捕获的 origin turn_id）。

2. `adapter/platform_event.py` 与 `adapter/platform_adapter.py`
- 若采用 inline 字段通道，需要恢复字段透传；若采用标签方案可不改。

3. `adapter/adapter/protocol.py`
- 增加 `engine.motion_plan` payload 最小 schema 校验，避免脏数据进入主链路。

4. `frontend/src/composables/useAdapterConnection.ts`
- `applyInboundMotionPlan()` 增加 `turn_id` 守卫，拒绝过期轮次计划。

## AstrBot 约束与模式结论

本方案保持你要求的“仅 AstrBot provider 模式”：
- 不引入 OpenAI/Ollama 直连新分支。
- OpenAI 仅作为 AstrBot 的 provider 选择，不改变项目连接边界。

## 风险清单与缓解

- JSON 不稳定：严格解析 + 失败回退。
- 模型漂移：为动作块增加 schema_version 与字段白名单。
- 安全边界：debug API 增加开关与鉴权，避免外部注入测试计划。

## 建议决策

建议按“方案 A + 分阶段切换”推进，先做影子验证再切主，能在不破坏现有功能的前提下，最快验证你想要的“单次请求低延迟动作链路”。
