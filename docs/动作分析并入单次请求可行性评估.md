# 动作分析并入单次请求

快照日期：2026-04-23

范围：AstrBot Provider 模式下，主聊天回复同时产出文本与 `<@anim {...}>`，adapter 在同一轮内提取并广播 `engine.motion_plan`。

## 当前实现

- 主请求注入：`adapter/adapter/turn_coordinator.py`
  - `_commit_inbound_message()`
  - `_apply_inline_motion_contract_to_event()`
  - `_build_model_visible_user_text()`
- 主回复提取：`adapter/adapter/turn_coordinator.py`
  - `emit_message_chain()`
  - `_extract_inline_motion_plan()`
- 二次请求兜底：`adapter/adapter/turn_coordinator.py`
  - `_schedule_realtime_motion_plan_preview()`
  - `_generate_and_broadcast_realtime_motion_plan()`

## 主链路

1. 用户消息进入 adapter。
2. `TurnCoordinator` 在提交 AstrBot 事件前改写 `event.message_str`。
3. 注入内容使用 `<system_reminder>` 包裹内联动作契约。
4. 主模型先输出正常回复文本。
5. 主模型最后一行输出单个 `<@anim {...}>`。
6. adapter 从回复中剥离 `<@anim {...}>`。
7. 文本继续走 `output.text`。
8. 动作部分校验为 `engine.parameter_plan.v1` 后广播 `engine.motion_plan`。

## 兜底链路

- 条件：主回复没有 `<@anim {...}>`，或标签解析失败，或动作计划校验失败，或内联计划下发失败。
- 行为：触发 `realtime_motion_plan` 二次请求。
- 目标：尽量减少因 provider 网络抖动导致的“完全无动作”概率，但不再作为首选路径。

## 相关配置

- `enable_inline_motion_contract`
  - 类型：`bool`
  - 默认：`true`
  - 作用：是否向主聊天请求注入 `<@anim ...>` 输出契约。
- `enable_realtime_motion_plan`
  - 类型：`bool`
  - 默认：`true`
  - 作用：主回复未产出合法内联动作计划时，是否继续走二次请求兜底。
- `realtime_motion_timeout_seconds`
  - 类型：`float`
  - 默认：`8.0`
  - 作用：二次请求兜底超时时间。

## 输出约束

- `<@anim {...}>` 必须是回复最后一行唯一内容。
- 标签内 JSON 必须为合法 JSON。
- 顶层字段使用：
  - `mode`
  - `plan`
- `plan.schema_version` 固定为 `engine.parameter_plan.v1`。
- `plan.mode` 允许 `idle` 或 `expressive`。
- `plan.key_axes` 必须包含 12 个直写参数轴。

## 现阶段结论

- “主聊天注入 + 主回复提取”已经落地。
- “回复后二次请求”现在是 fallback，不再是主链路。
- 当前剩余风险主要在前端 Live2D 执行稳定性，不在动作计划注入/提取路径本身。
