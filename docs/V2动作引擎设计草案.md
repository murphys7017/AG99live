# V2 动作引擎设计草案

本文用于把 AG99live V2 的动作引擎设想，从“方向判断”进一步细化到“可以逐步开工的设计草案”。

当前不追求把所有细节一次性定死，而是先把以下内容讲清楚：

1. 动作引擎到底要解决什么问题
2. 它的输入和输出分别是什么
3. 它应该拆成哪些模块
4. 第一阶段最小可用版本应该先做什么
5. 哪些问题先不做，避免一开始就失控

## 1. 动作引擎要解决的核心问题

V1 的主要问题是：

- 对一整段回复，通常只挑一个表情或一个动作
- 很多表现方式是静态映射
- 面对不同语音时长、节奏和强调点时，动作显得机械
- 同一句语义多次出现时，表现很容易重复

因此，V2 动作引擎的核心问题不是“怎么多播几个动作”，而是：

**如何把一段回复的语义、音频长度、节奏和情绪，转成一段连续、可变化、可回退的 Live2D 实时表现。**

这意味着动作引擎本质上是一个：

- 运行时调度器
- 参数化动画组合器
- fallback 决策器

而不是简单的“动作文件选择器”。

## 2. 动作引擎的总体目标

动作引擎的总体目标可以拆成四条：

1. 让回复表现不再完全依赖人工写死的 `motionMap`
2. 让同一种语义可以表现出多种不同但仍然合理的动作
3. 让动作表现能随着音频时长、节奏、强调点发生变化
4. 在不同模型能力差异很大的情况下，仍然可以工作，并且能够平稳降级

## 3. 动作引擎不做什么

为了避免一开始设计过重，先明确它不做什么：

- 不直接生成新的 `.motion3.json` 文件
- 不追求一开始就覆盖所有模型参数
- 不让大模型直接逐帧控制动作
- 不要求第一阶段就达到“非常拟人”的表现
- 不要求第一阶段就把所有 UI 交互都接完

一句话：

**第一阶段的动作引擎是“能稳定运行并能比 V1 更灵活”，不是“马上做到极致动画系统”。**

## 4. 动作引擎的输入

动作引擎接收的输入可以分成四类。

### 4.1 后端同包输入

这是每次回复最直接的输入来源。

这类输入建议由后端在同一次回复中一并带给前端，而不是让前端再额外请求一轮。

这类输入的语义层面可能包括：

- 回复文本
- 音频 URL 或音频时长信息
- 基础情绪倾向
- 基础风格倾向
- 强调度或能量级别
- 可选的动作提示或模板线索

这里最重要的原则是：

- 后端负责给低频语义输入
- 前端负责把这些输入翻译成运行时动作

### 4.2 模型能力输入

这部分来自模型扫描结果或基础 manifest。

它回答的问题包括：

- 当前模型有哪些参数
- 哪些参数可以被实时驱动
- 有哪些 expression
- 有哪些 motion
- 哪些 motion 可以作为 fallback
- 哪些参数范围是安全的

### 4.3 运行时上下文输入

这部分来自前端当前状态。

例如：

- 当前是否正在播放音频
- 当前音频时长
- 当前播放进度
- 当前是否已经有表情占用
- 当前是否已经在执行某个 fallback motion
- 当前桌宠模式状态

### 4.4 配置输入

这是引擎本身的行为配置，不是单次回复数据。

例如：

- 默认动作模板
- 各通道的基础强度
- 随机扰动范围
- fallback 开关
- 动作切段阈值
- 长句和短句的节奏策略

## 5. 动作引擎的输出

动作引擎的输出不是新动作文件，而是运行时执行结果。

可以分成三类：

### 5.1 参数输出

这是最核心的输出。

例如：

- 头部角度偏移
- 身体轻微摇摆
- 眼睛开合偏置
- 眉毛抬起或压低
- 嘴角笑意偏置
- 视线朝向

这些输出最终会被写进 Live2D 参数。

### 5.2 expression 输出

当某个阶段更适合用表情文件表达时，引擎可以输出 expression 切换指令。

这类输出更适合：

- 快速建立基础情绪
- 在参数能力不足的模型上做稳定回退

### 5.3 motion fallback 输出

当模型本身已有一组高质量 motion 且更适合直接使用时，引擎可以输出 motion 调用决策。

这类输出不应是主路径，但它非常重要，因为它能让动作引擎在不同模型上更稳。

## 6. 动作引擎的推荐模块拆分

建议把前端动作引擎拆成以下模块。

## 6.1 Engine Coordinator

这是主调度器。

职责：

- 接收单次回复输入
- 创建本轮动作任务
- 调用各子模块
- 驱动更新循环
- 在播放结束后做收尾

它不直接做参数计算，而是负责组织流程。

## 6.2 Capability Resolver

职责：

- 读取当前模型能力信息
- 判断当前模型支持哪些通道
- 决定哪些通道可实时驱动
- 决定是否需要更多依赖 expression 或 motion fallback

这个模块的意义是让引擎适应不同模型差异。

## 6.3 Semantic Planner

职责：

- 根据后端输入和回复内容，得到本轮动作语义
- 例如：
  - `style = explain`
  - `mood = warm`
  - `energy = 0.45`
  - `emphasis = 0.7`

它产出的不是具体参数值，而是动作意图。

## 6.4 Timeline Planner

职责：

- 根据音频时长和节奏，把一轮回复拆成若干阶段
- 决定：
  - 开场阶段
  - 中段阶段
  - 强调阶段
  - 收尾阶段

它解决的是“动作何时发生、分几段发生”的问题。

这是 V2 的关键模块之一。

## 6.5 Channel Mixer

职责：

- 把语义和时间轴翻译成具体通道值
- 将多个通道组合起来
- 控制每个通道的强度、频率、偏移量
- 处理随机扰动和基础模板叠加

它可以理解成动作引擎的“实时合成器”。

## 6.6 Fallback Resolver

职责：

- 当参数驱动不够时，判断是否切换 expression
- 当参数能力不足时，判断是否调用 motion
- 在参数驱动、expression、motion 之间做平稳过渡

这个模块对多模型兼容非常关键。

## 6.7 Runtime Driver

职责：

- 在更新循环中逐帧执行参数写入
- 驱动 expression 切换
- 驱动 motion 调用
- 和音频播放状态同步

这个模块是最靠近渲染层的执行器。

## 7. 推荐的更新循环

动作引擎的推荐运行方式是逐帧更新，而不是每次回复只做一次静态决策。

建议逻辑如下：

1. 收到本轮回复输入
2. Engine Coordinator 创建动作任务
3. Semantic Planner 生成动作语义
4. Timeline Planner 生成时间轴
5. Channel Mixer 初始化通道状态
6. Runtime Driver 在每帧更新中：
   - 读取当前播放进度
   - 计算当前阶段
   - 计算各通道当前值
   - 写入模型参数
   - 必要时触发 expression 或 fallback motion
7. 播放结束后执行收尾和状态恢复

## 8. 第一阶段建议先支持的参数通道

当前阶段不建议一下子把参数通道铺太多。

第一阶段只建议优先做一小组高价值、相对通用、最能提升观感的通道：

- `headYaw`
- `headPitch`
- `bodySway`
- `gazeX`
- `gazeY`
- `eyeOpenBias`
- `browBias`
- `mouthSmileBias`
- `breathBias`

这一组通道已经足以让表现明显区别于 V1 的“静态动作映射”。

后续再逐渐扩展：

- headRoll
- cheek / blush
- shoulder / arm related params
- accessory sway related params

## 9. 第一阶段的模板思路

第一阶段不需要很多模板，但至少建议先有几种基础模板：

- `idle`
- `warm-talk`
- `explain`
- `thinking`
- `happy`
- `surprised`
- `emphasis`

模板的作用不是直接输出最终动作，而是提供：

- 通道默认权重
- 强度上下限
- 常用节奏
- 通道偏向

例如：

- `thinking`
  - 更偏头部小幅停顿和眼神偏移
- `happy`
  - 更偏嘴角笑意、轻微头部活跃度
- `explain`
  - 更偏平稳 body sway 和轻度强调动作

## 10. 第一阶段的切段策略

为了避免一开始过难，第一阶段建议用简单分段策略，而不是复杂文本韵律分析。

例如按音频时长粗分：

- 短句：1 段主动作 + 收尾
- 中句：开场 + 中段 + 收尾
- 长句：开场 + 中段 + 强调 + 收尾

再根据标点或强调度，决定是否插入一个 emphasis 脉冲。

这样已经能比“整句一个 motion”好很多。

## 11. 第一阶段推荐的 fallback 策略

动作引擎第一阶段一定要接受“并不是所有模型都适合全参数驱动”。

因此建议 fallback 策略明确分级：

### Level 1

优先参数驱动，辅以 expression。

### Level 2

参数驱动能力不足时，使用 expression 作为主情绪承载。

### Level 3

模型参数能力更弱时，用 motion 作为增强或主表现方式。

### Level 4

极端情况下只保留基础 expression + lip sync，也要保证稳定。

这套 fallback 机制会让 V2 真正具备多模型通用性。

## 12. 第一阶段最小可用版本

动作引擎的第一阶段 MVP 建议非常克制，只验证主路径。

MVP 目标：

- 固定一个测试模型
- 使用手写或 mock 的能力配置
- 跑通基础模板
- 跑通音频时长驱动的简单切段
- 跑通 5 到 9 个基础参数通道
- 跑通 expression fallback
- 跑通至少一种 motion fallback

只要这条链路跑通，V2 的核心方向就算被验证了。

## 13. 当前阶段先不要做的事

为了控制复杂度，当前阶段建议先不做：

- 完整自动文本节奏分析
- 复杂 NLP 级别的动作规划
- 大量 UI 配置面板
- 对所有模型做完整兼容
- 所有 motion 的自动语义理解
- 非常复杂的参数组联动

这些事情未来都可以做，但不应该成为第一阶段的阻塞点。

## 14. 当前结论

当前已经明确的动作引擎方向如下：

- 动作引擎是 V2 前端最难、最重要的核心
- 它本质上是运行时调度器 + 参数化动画组合器 + fallback 决策器
- 后端负责给低频输入，前端负责生成实时动作表现
- 第一阶段优先做小而稳的参数通道和简单时间轴
- 第一阶段必须内建 fallback 机制
- 第一阶段的成功标准不是“很复杂”，而是“主路径可运行且明显优于 V1”

一句话总结：

**V2 动作引擎的正确方向不是“多播几个动作”，而是“建立一个能逐帧生成动作表现的前端 runtime”。**
