# 二期架构方向草案

本文用于记录当前已经达成共识的二期方向。它不是功能说明文档，而是一份架构决策草案，用来回答三个问题：

1. V2 到底准备解决什么问题
2. V1 和 V2 的边界有什么本质变化
3. 是否应该新开一个项目来承载 V2

同时，这份文档也承担一个额外目的：

**把 V2 视为长期工程，并尽量把当前已经想清楚的关键判断固化下来，避免后续开发过程中再次退回到 V1 的思路里。**

## 1. 背景

当前 V1 方案已经完成了插件端与桌面端前端的基础联通，具备以下能力：

- AstrBot 插件作为平台适配器，提供 WebSocket 和静态资源服务
- 桌面端前端加载 Live2D 模型并完成文本、音频、表情、动作的基本联动
- 后端根据 `emotionMap`、`motionMap`、`motion_catalog.json` 选择已有动作或表情
- 前端负责播放音频、口型同步、字幕展示和基础动作触发

V1 的主要问题也已经比较明确：

- 当前动作驱动方式偏静态，主要还是“为整段回复挑一个表情或动作”
- 手工维护 `motionMap` / `emotionMap` 的成本较高，换模型后复用性差
- 一旦考虑语音长短、节奏、强调点、连续对话，固定动作映射会显得机械
- 插件后端目前承担了过多本应属于本地桌面运行时的职责
- 当前前端项目虽然能继续演进，但它的定位已经不再是“上游网页前端改造版”，而更接近“本地 Live2D 桌宠运行时”

因此，V2 不再是简单追加功能，而是一次职责重划和运行时升级。

## 1.1 一个必须持续提醒自己的事实

V2 不是“把 V1 做大一点”，而是“重新定义整个桌面端的定位”。

这句话需要在整个项目周期内持续成立：

- V1 更像对既有项目的场景化改造
- V2 更像基于既有经验重新设计的新系统

因此，V2 后续任何设计都应优先问自己：

- 这是在延续旧前端思路，还是在建设新的桌面 runtime
- 这是在补一个临时兼容逻辑，还是在完善长期架构
- 这是在为单一模型写特例，还是在为多模型通用能力建设底座

如果答案偏向前者，就需要警惕方向漂移。

## 2. V2 的核心目标

V2 的目标不是“多录一点动作”，而是建立一套更通用的 Live2D 动作系统，使其能够：

- 支持多种不同结构的 Live2D 模型
- 自动扫描模型能力，而不是完全依赖手工配置
- 基于基础模板和参数范围进行实时动作编排
- 根据语音时长、节奏、情绪和强调点动态生成更自然的动作表现
- 让 UI 框架和 Live2D runtime 解耦，便于后续扩展网关模块和桌面交互能力

一句话总结：

**V2 的目标是把当前桌宠前端升级为一个本地 Live2D 实时动作引擎，而不是继续把它当成一个普通网页前端。**

## 3. 当前确定的方向

### 3.1 插件后端的定位

AstrBot 插件端在 V2 中继续保留，但职责应当收缩为：

- AstrBot 平台适配器
- 模型与静态资源提供者
- 基础配置入口
- 与 AstrBot 对话链路的协议桥接

插件端不再继续膨胀成“动作总控中心”。尤其是以下职责，不应继续以插件后端为主要落点：

- 运行时动作编排
- 实时参数计算
- 音频播放时序驱动
- 本地桌面交互逻辑

插件端可以保留部分低频分析能力，例如：

- 模型资源扫描
- 能力清单生成
- 默认模板生成
- 安全范围或建议范围导出

但这些更像“分析产物生成”，而不是“实时执行逻辑”。

### 3.2 前端的定位

V2 中的前端不再只是“UI + WebSocket 客户端”，而是：

**本地 Live2D 实时动作引擎 + 用户交互壳层**

它需要承担：

- Live2D 模型加载
- 参数级动作驱动
- 表情切换
- 音频与口型同步
- 动作时间轴编排
- motion / expression fallback
- 本地设备桥接
- UI 和网关子模块接入

更准确地说，前端 runtime 的产出主要不是新的 `.motion3.json` 文件，而是：

- 实时参数曲线
- expression 切换指令
- motion 调度结果
- 运行时 fallback 决策

## 4. 动作系统的设计原则

### 4.1 不再只依赖手工 motionMap

手工配置仍然保留，但只作为：

- 默认模板
- 人工修正入口
- 特殊模型兜底方案

不再把它当作唯一动作来源。

### 4.2 建立“模型能力 -> 运行时动作”的两层结构

建议将动作系统拆成两层：

1. 模型能力层
2. 运行时编排层

#### 模型能力层负责回答

- 这个模型有哪些参数
- 哪些参数适合被实时驱动
- 哪些表情文件可用
- 哪些 motion 文件可用
- 哪些 motion 主要影响头、身体、眼、嘴、手
- 各参数适合的安全范围和推荐幅度

#### 运行时编排层负责回答

- 当前回复应该偏什么风格
- 动作强度应该多大
- 语音时长下应该切几段
- 哪些时间点加入强调动作
- 当前该优先走参数驱动还是直接走 motion fallback

### 4.3 运行时以“参数驱动”为主，motion 为辅

长期目标不是为每句话生成一个新的 motion 文件，而是：

- 用参数驱动形成头部、身体、眼睛、眉毛、嘴部等通道的实时运动
- 用已有的 motion / expression 作为增强层或 fallback

也就是说，V2 更像一个“参数化动画系统”，而不是一个“动作文件选择器”。

### 4.3.1 动作系统的核心观念

后续开发动作系统时，必须反复坚持以下观念：

- 不再把“挑一个 motion 来播完一整句”当成主路径
- 不再把“多录几个动作文件”当成主要优化方向
- 不再把后端决定最终动作细节当成默认方案

取而代之的是：

- 先理解模型具备哪些可控能力
- 再抽象出统一的动作语义和参数通道
- 最后由前端 runtime 在播放时实时生成动作表现

动作系统的终局目标不是“维护一份越来越大的动作映射表”，而是：

**建立一套能够在不同模型之间迁移的参数化动画方法。**

### 4.3.2 动作系统的推荐分层

建议把 V2 动作系统理解成以下五层：

1. 资源层
   - `model3.json`
   - `cdi3.json`
   - `physics3.json`
   - `exp3.json`
   - `motion3.json`

2. 能力层
   - 扫描可用参数
   - 扫描可用 expression / motion
   - 判断哪些参数适合实时驱动
   - 判断哪些 motion 适合作为 fallback 或增强

3. 语义层
   - 定义统一语义，例如：
     - `idle`
     - `greet`
     - `explain`
     - `thinking`
     - `happy`
     - `surprised`
     - `emphasis-small`
     - `emphasis-large`

4. 编排层
   - 根据文本、音频时长、情绪、强调点、节奏，生成一段时间轴计划
   - 决定每个阶段使用哪些参数通道、强度多大、是否插入 fallback motion

5. 执行层
   - 在前端逐帧写入 Live2D 参数
   - 切换 expression
   - 调用 motion fallback
   - 与口型、字幕、音频状态同步

这五层中，最重要的是中间三层：能力层、语义层、编排层。它们才是 V2 的真正核心资产。

### 4.3.3 参数化动作系统的最小目标

V2 初期不必追求“所有模型都极其生动”，但至少要做到：

- 能扫描并识别模型的基础参数能力
- 能建立一组跨模型通用的基础参数通道
- 能根据回复长度和情绪强度生成不同的动作节奏
- 能让同一种语义不再每次都表现得完全一致
- 能在参数驱动失败或模型能力不足时平稳回退到 expression / motion

也就是说，V2 的最小可用标准不是“动作很多”，而是：

**动作表现开始由系统实时生成，而不是完全依赖人工预先写死。**

### 4.4 不让大模型直接控制每一帧

模型或规则层只适合输出低频语义，例如：

- 当前风格：`explain`
- 当前情绪：`warm`
- 当前能量：`0.45`
- 当前强调度：`0.7`

随后由前端 runtime 将这些低频语义翻译成实时参数变化。

这样做的原因是：

- 更稳定
- 更可控
- 更通用
- 更利于多模型兼容

### 4.5 动作系统需要长期沉淀，而不是一次做完

这套系统天然是长期工程，后续一定会经历以下阶段：

1. 先跑通基础参数通道
2. 再建立多模型扫描和 manifest
3. 再建立语义模板和编排器
4. 再慢慢提高动作多样性和自然度
5. 最后再考虑更复杂的自动生成、风格化和个性化

因此，开发过程中要刻意避免两种心态：

- 试图一次性做完一个“完美动作引擎”
- 因为短期效果一般，就退回到继续堆 motionMap 的旧方案

正确做法是持续积累：

- 扫描能力
- 参数通道
- 模板库
- 编排规则
- fallback 策略
- 多模型经验

## 5. V2 推荐的职责拆分

### 5.1 插件端

建议保留：

- AstrBot 平台适配与协议桥接
- 模型资源目录管理
- 静态资源服务
- 模型扫描入口或分析任务触发
- 默认 manifest / 模板分发

建议逐步迁出：

- 表情与动作的最终决策
- 对播放时机强依赖的动作逻辑
- 本地运行时状态机

### 5.2 桌面端 runtime

建议新增并集中在这里：

- `model-scan`
- `capability-manifest`
- `animation-runtime`
- `parameter-scheduler`
- `motion-fallback`
- `audio-sync`
- `platform-bridge`

## 6. UI 框架与 Live2D runtime 解耦

当前项目使用 React，但这不是 V2 的核心约束。

已经确定的原则是：

- 继续使用 `Electron`
- 继续使用 `TypeScript`
- Live2D runtime 必须与 UI 框架解耦
- UI 层可以从 React 迁移到 Vue

因此，后续如果需要改成 Vue，正确的顺序应当是：

1. 先把 Live2D runtime、协议层、音频层、平台桥接层从 React hooks/context 中抽离出来
2. 将它们重构为框架无关的 TypeScript 模块
3. 再使用 Vue 3 作为新的 UI 层进行接入

这样可以避免“重写 UI”和“重写动作系统”同时发生。

## 7. 是否需要新开项目

结论：

**建议新开项目，但不是立刻废弃 V1。**

原因如下。

### 7.1 为什么建议新开项目

当前 V1 项目和我们已经确定的 V2 目标相比，已经存在较明显的定位偏差：

- V1 是“基于现有 Open-LLM-VTuber 前端改造的 AstrBot 桌宠前端”
- V2 是“面向多模型通用的本地 Live2D 实时动作 runtime”

这两者虽然连续，但工程重心已经不同。

如果继续在现有 V1 项目中直接叠加，容易出现：

- 历史兼容逻辑越来越重
- React 结构和 runtime 结构纠缠更深
- 文档命名、目录命名、模块边界持续混乱
- 后续难以形成稳定的桌面 runtime 产品化结构

### 7.2 为什么不是马上重写一切

虽然建议新开项目，但不建议一上来全量推倒。更稳的策略是：

1. V1 继续保持可运行
2. 将 V2 作为新项目立项
3. 从 V1 中迁移可复用部分：
   - Electron main / preload 经验
   - Live2D Web SDK 接入方式
   - 音频与口型同步链路
   - 协议兼容逻辑中有价值的部分
4. 在新项目中重建更干净的 runtime 边界

### 7.3 建议的新项目定位

建议 V2 新项目的定位明确为：

**AstrBot Desktop Runtime / Live2D Runtime**

它不再只是“某个插件的网页前端”，而是一个：

- 本地桌面运行时
- 支持 Live2D 动画引擎
- 可承载未来网关子模块
- 可作为 AstrBot 的桌面客户端或桌宠 runtime

插件仓库则继续保持“适配器插件”的定位。

## 8. 推荐的后续实施顺序

### 第一阶段：立项与边界整理

- 确定 V2 新项目名称和目录结构
- 明确插件端与桌面端的协议边界
- 先定义 manifest、能力清单、模板数据结构

### 第二阶段：抽离 runtime 核心

- 从当前前端中抽离 Live2D runtime
- 抽离音频同步与动作调度
- 抽离 Electron bridge

### 第三阶段：建立模型能力系统

- 扫描 model3 / cdi3 / physics3 / exp3 / motion3
- 生成 capability manifest
- 建立参数通道映射和安全范围

这一阶段的重点不是“让模型动起来”，而是沉淀以下长期资产：

- 模型能力描述格式
- 参数分类方法
- motion / expression 的自动归类方法
- 通用参数通道和范围定义

### 第四阶段：建立实时动作编排器

- 基础模板
- 参数化动作通道
- 强度控制
- 时长缩放
- emphasis 插入
- motion fallback

这一阶段是 V2 的核心工程，后续绝大多数体验提升都来自这里，而不是来自继续追加单个动作文件。

### 第五阶段：Vue UI 接入

- 设置页
- 调试页
- 消息面板
- 模型面板
- 网关子模块入口

## 9. 当前阶段的明确结论

当前已经确定的结论如下：

- V2 不再继续沿着“多录动作、多写 motionMap”的方向发展
- V2 的核心是“多模型通用的参数化实时动作系统”
- AstrBot 插件应逐步回归适配器角色
- 前端应升级为本地 Live2D 实时动作引擎
- 技术栈暂定为 `Electron + TypeScript + Vue`
- Live2D runtime 必须与 UI 框架解耦
- 建议为 V2 新开项目，并逐步迁移 V1 中可复用的实现

这意味着，V2 与 V1 的差异已经不仅是“功能增量”，而是“项目定位升级”。

## 10. 给未来开发中的自己

如果后续开发中出现摇摆，可以先回到下面这些判断：

- V2 的核心不是“前端页面重写”，而是“桌面 runtime 重建”
- V2 的核心不是“继续适配一个模型”，而是“建立多模型通用能力”
- V2 的核心不是“把动作配置写得更细”，而是“建立参数化实时动作系统”
- V2 的核心不是“让后端继续做更多事”，而是“让插件回归适配器，让前端承担实时引擎职责”
- V2 的核心不是“追求一次到位”，而是“先把正确的分层和演进路线立住”

如果以后某个方案虽然短期方便，但会让项目重新退回：

- 单模型特化
- 大量手工 motionMap
- 后端主导动作细节
- UI 和 runtime 再次缠死

那就应该优先谨慎，哪怕短期慢一点，也不要轻易回退。
